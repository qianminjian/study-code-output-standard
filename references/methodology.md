# Method Reference — 方法论摘要

> **按需加载**：这是 skill 触发的 6 步法背后的完整方法论摘要。
> Claude Code 在 Step 1 之前 Read 本文件，作为方法论基础。

## 1. 三大核心原则

### 1.1 资产分层（5 层）

```
L1 导航层    00-索引
L2 全局层    01-总览
L3 静态层    02-数据 / 03-API / 04-Mapper / 05-服务 / 06-安全 / 07-页面 / 08-状态 / 09-多端
L4 行为层    10-业务流
L5 诊断层    11-技术债 / 12-修复
```

### 1.2 资产可消费（双受众）

- **人**：自然语言 + 表格 + 流程图
- **AI**：严格的 Markdown 结构 + 编号 + 标签 + 元信息头

### 1.3 资产可演进

- 元信息头 7 字段
- 严重度 P0-P3
- 编号稳定
- 配套 CI 校验

## 4. 严重度等级（统一）

| 等级 | 含义 | 修复 SLA | 标识 |
|---|---|---|---|
| 🔴 P0 | 安全漏洞 / 数据丢失 / 系统不可用 | 24 小时内 | 必修 |
| 🟡 P1 | 功能缺陷 / 体验问题 / 性能 | 1 季度内 | 必排 |
| 🟢 P2 | 架构 / 一致性 / 可维护性 | 半年内 | 计划 |
| ⚪ P3 | 死代码 / 命名 / 注释清理 | 顺便 | 可缓 |

**写入规则**：
- 严禁使用 P4、P5 或自定义
- 跨表/跨文件用同一严重度

## 5. 6 步法详细

### Step 1 · 勘探（Reconnaissance）— 1-2 天

**目标**：建立项目"心智地图"

**必做动作**：
- [ ] 跑项目主干（按 01-总览的"启动顺序"）
- [ ] 用 `cloc` / `tokei` 统计代码量
- [ ] 用 `tree -L 3` 拿顶层目录
- [ ] 列 `package.json` / `pom.xml` 拿依赖
- [ ] 找到入口文件

**输出**：项目骨架笔记（半页纸）

### Step 2 · 分类（Classification）— 0.5 天

**目标**：映射到 12 篇资产框架

**决策表**：

| 现象 | 决策 |
|---|---|
| 有 `views/` 和 `components/` | 07 完整输出 |
| 静态 HTML 站 | 09 完整输出 |
| 后端用 Spring Boot + MyBatis | 04 完整输出 |
| 前端用 Vue 2 | 08 含 Vuex |
| 前端用 Vue 3 | 08 含 Pinia |
| 移动 App | 09 拆 09-iOS / 09-Android |
| 微服务多仓 | 加 13-跨服务索引 |

### Step 3 · 抽取（Extraction）— 3-7 天

**目标**：从代码生成 12 篇资产初稿

**关键原则**：**先静态后行为，先具体后综合**

```
1. 02-数据模型 ← ORM 实体 + DDL（最具体）
2. 03-API 清单 ← Controller / Route（最稳定）
3. 04-Mapper 清单 ← 依赖 02
4. 06-安全认证 ← 单独扫（优先级高）
5. 05-服务与业务逻辑 ← 依赖 02/03/04
6. 07-前端页面与组件 ← 单独扫
7. 08-前端状态与路由 ← 依赖 07
8. 09-静态站 ← 单独扫
9. 10-业务流图 ← 跨 02-09 串联
10. 11-技术债 ← 跨 01-10 累积
11. 12-修复建议 ← 派生 11
12. 01-系统总览 ← 总结所有
13. 00-文档索引 ← 最后写
```

### Step 4 · 校验（Validation）— 1-2 天

**目标**：确保资产与代码事实一致

**三类校验**：
- **自动**（CI / 脚本）：端点数、文件数、字段完整性
- **抽样**（人）：5-10 个端点、3-5 个业务流
- **同行**（另一人）：是否漏关键资产

### Step 5 · 标注（Annotation）— 1-2 天

**目标**：把代码的"问题"与"风险"显式标出

**三类标注**：
- 严重度标注（必备）
- 位置标注（`<文件>:<行号>` 格式，必备）
- 反模式标签（可选但推荐）

### Step 6 · 派送（Delivery）— 0.5 天

**目标**：让资产可发现、可消费、可演进

**必做动作**：
- [ ] 写 `${TARGET}/CLAUDE.md`（轻量索引 + 按需加载标注）
- [ ] 写 `${TARGET}/CLAUDE-ASSET.md`（资产详情）
- [ ] 接 CI：`docs/scripts/validate-all.sh` 在 PR 必跑
- [ ] 发布到 Backstage / GitBook / VitePress（如有内部平台）
- [ ] 通知团队

## 6. 强引用关系（必须保持一致）

| 关系 | 检查方式 |
|---|---|
| 02 ↔ 03 | 03 提到 X 实体 → 02 必有 X 表 |
| 02 ↔ 04 | 实体 ↔ Mapper |
| 03 ↔ 05 | API ↔ Service |
| 03 ↔ 06 | API ↔ 鉴权 |
| 07 ↔ 08 | 组件 ↔ 状态/路由 |
| 10 ↔ 02-09 | 业务流 ↔ 各层资产 |
| 11 ↔ 02-10 | 技术债 ↔ 各层资产 |
| 12 ↔ 11 | 修复建议 ↔ 技术债 |

## 7. 适用边界

**✅ 适用**：存量项目接手 / 老系统现代化 / AI 编程上下文建设 / 新人入职 / 架构治理

**❌ 不适用**：全新项目从零起步 / 1-2 周 Demo / 单一库/SDK

**🟡 部分适用**：微服务多仓 / 纯前端 SPA / 移动 App（需做变体）

## 9. CI 接入（GitHub Actions 示例）

```yaml
# .github/workflows/docs-validate.yml
name: Docs Validate
on:
  pull_request:
    paths:
      - 'asset-docs/**'
      - 'src/**'
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run all checks
        run: bash asset-docs/scripts/validate-all.sh
```

## 11. 关键金句

> 改代码 = 改资产；改资产 = 改代码。资产是代码的伴生品而非附件。

> P0 > 10 条 = 没分级。严重度体系失效比没体系更危险。

> AOP 切面是最容易被遗忘的"半成品"——代码在，但永远不触发。

### 11.1 闭环机制（

为让"**改代码 = 改资产**"金句真正落地，本 skill 引入 **Step 7 · 资产回写**：

| 触发 | 动作 | SLA |
|---|---|---|
| PR 改 Controller | 同步 03-Controller + 12 修复建议（如有影响） | PR 合并前 |
| PR 改 Mapper | 同步 02-数据模型 + 04-Mapper | PR 合并前 |
| PR 修 P0 | 11 标"已修复" + 12 标"已完成" | PR 合并前 |
| 资产 `last_updated` > 30 天 | `check-meta.sh` 告警 | 不阻断，建议 owner 刷新 |
| 代码 bump 大版本 | 12 篇 `version` 同步 bump + CHANGELOG | 发版前 |

**配套 CI**：

```yaml
# .github/workflows/docs-validate.yml
on:
  pull_request:
    paths: ['src/**', 'asset-docs/**']
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: bash $SKILL_HOME/scripts/check-all.sh
```

> 闭环失败 = 资产漂移 = 失去 AI 喂入价值。**第一次漂移出现就要抓**，不要等季度 review。
