## 完整 Prompt

```
【任务背景】
你是一个资深的反向阅读专家。
我的项目 <PROJECT_NAME>（代码根目录：<PROJECT_PATH>）需要按"study-code-output-standard"方法论
反向生成 12 篇标准资产。

【核心原则】
⚠️ 不按模板预设清单输出，按代码实际扫描结果输出！
- 有哪些就列哪些，没有就不写
- 章节编号从实际出发，不是从模板固定编号出发
- 所有数字从代码统计，不是预设值

【方法论位置】
方法论文档位于 references/methodology/（或外部路径 <METHODOLOGY_PATH>）。
请先读完以下 4 篇再开始：
1. references/methodology/00-总览与方法论.md
2. references/methodology/01-资产清单与适用场景.md
3. references/methodology/02-目录与命名规范.md
4. references/methodology/04-反向阅读工作流.md

【输出位置】
所有 12 篇资产输出到 <OUTPUT_DIR>/（默认 ./asset-docs/）。

【模板位置】
12 份可填充模板位于 <TEMPLATE_DIR>/（默认 ./asset-docs/assets/）。
每篇资产先复制对应模板，再填内容。

【AI Prompt 工具集】
每个资产的详细 Prompt 在 <PROMPT_DIR>/（默认 ./asset-docs/references/prompts/）下的 01-12.md。
请按 04-工作流 §3.1 的顺序，**逐个调取**这些 Prompt 来生成对应资产。

【6 步法执行流程】
严格按 references/methodology/04-反向阅读工作流.md 执行：

### Step 1 · 勘探（自动）
跑 `tree -L 3 <PROJECT_PATH>` 看目录。
读 README、package.json / pom.xml / go.mod / requirements.txt。
找 5 个最关键的入口文件。
输出：项目骨架笔记（1 页）。

### Step 2 · 分类（自动）
套用 references/methodology/01-资产清单与适用场景.md 的适用矩阵。
决定：12 篇资产哪些输出 / 哪些占位 / 哪些省略。
不适用也保留编号（如纯后端项目 07/08/09 写"本项目无前端"）。

### Step 3 · 抽取（核心）
按 04-工作流 §3.1 的依赖顺序生成 12 篇资产：

1. 02-数据模型与表结构（最具体）→ 调 02-数据模型与表结构.md Prompt
2. 03-API 清单 → 调 03-后端-Controller接口清单.md Prompt
3. 04-Mapper 清单 → 调 04-后端-Mapper操作清单.md Prompt
4. 06-安全认证 → 调 06-后端-安全认证.md Prompt
5. 05-服务与业务 → 调 05-后端-服务与业务逻辑.md Prompt
6. 07-前端页面（如适用）→ 调 07-前端-页面与组件清单.md Prompt
7. 08-前端状态（如适用）→ 调 08-前端-状态管理与路由.md Prompt
8. 09-静态/多端（如适用）→ 调 09-静态前台.md Prompt
9. 10-业务流图（跨层串联）→ 调 10-业务流图.md Prompt
10. 11-技术债（跨资产综合）→ 调 11-技术债与遗留项.md Prompt
11. 12-修复建议（派生 11）→ 调 12-修复建议与优先级.md Prompt
12. 01-系统总览（最后总结）→ 调 01-系统总览.md Prompt
13. 00-文档索引（最最后）→ 写索引页

每个资产**必须**包含：
- 顶部 frontmatter（id / version / last_updated / data_source / code_version / owner / ai_consumable）
- 顶部引用块 `> 整理时间 · 数据来源`
- 严格按 assets/ 对应文件结构
- 强制列填齐
- 严重度 P0-P3 标注
- 元信息表收尾

### Step 4 · 校验（自动 + 人工）
跑 <OUTPUT_DIR>/scripts/validate-all.sh。
修复脚本报告的所有错误。
人工抽 5-10 个端点对照 Swagger / Postman 验证。

### Step 5 · 标注（半自动）
跑 <OUTPUT_DIR>/scripts/scan-antipatterns.sh。
把扫出的反模式按 references/methodology/07-典型案例与反模式.md 的反模式标签体系
（@sqlinjection / @secret-leak / @cors-wildcard / @perm-mismatch / @n+1 / @xss / @hardcoded 等）
标注到 11-技术债清单。

### Step 6 · 派送（自动）
1. 写 <PROJECT_PATH>/CLAUDE.md（项目级 AI 引导），引用 asset-docs/ 的 12 篇资产
2. 在 <PROJECT_PATH>/.github/workflows/docs-validate.yml 加 CI
3. 在 <OUTPUT_DIR>/CHANGELOG.md 加 v1.0.0 条目
4. 通知团队"资产 v1.0 已就绪"

【自我校验（每篇资产生成后必跑）】
1. 跑 `wc -l <asset>.md` 不超 03-模板 §质量门禁
2. 跑 `grep -c <强制列> <asset>.md` 列数对齐
3. 跑 `bash asset-docs/scripts/check-meta.sh` 字段齐全
4. 跑 `bash asset-docs/scripts/check-severity.sh` P0 < 10
5. 端点数/Controller 数/Mapper 数 等数字与 grep 一致

【反例】
- 不写"约 70 个端点" 而 grep 数清楚
- 不写散文式段落，全部表格化
- 不写自定义严重度（"中"、"高"）
- 不漏 frontmatter 7 字段
- 密钥/密码必用 <REDACTED> 替代
- 不省鉴权列
- 不省严重度等级

【执行确认】
请回复：
1. 你的执行计划（6 步的细分步骤）
2. 需要的输入（项目路径、技术栈、是否前端/静态端等）
3. 预计耗时

确认后立即开始 Step 1。
```

## 使用方法

1. **复制本文件除【执行确认】段以外的所有内容**
2. **替换占位符**：
   - `<PROJECT_NAME>` → 项目名
   - `<PROJECT_PATH>` → 项目代码根目录的绝对路径
   - `<METHODOLOGY_PATH>` → 方法论位置（默认 `references/methodology/`）
   - `<OUTPUT_DIR>` → 资产输出目录（默认 `./asset-docs/`）
3. **粘贴到 AI 对话框**
4. AI 会反问几个问题（项目类型、技术栈等）
5. AI 按 6 步法自动执行

- [ ] 端点数字与代码 grep 一致
- [ ] 密钥/密码已 <REDACTED>
- [ ] CLAUDE.md 已写好
