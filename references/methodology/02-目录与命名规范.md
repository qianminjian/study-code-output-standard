# 02 · 目录与命名规范

> 这篇给所有"用本方法论的项目"用。**违反这一篇，模板与 AI Prompt 都会失配**。

## 2. 文件命名规则

### 2.1 编号

- **固定两位数字**（`00` 到 `12`），前缀 `0`
- 编号**严格不变**——不要为某个项目改成 `02a`、`02b`
- 项目级新增资产时用 `13+`（如 `13-跨服务索引.md`）

### 2.2 文件名格式

```
<两位编号>-<技术域>-<资产名>.md
```

| 部分 | 规则 | 示例 |
|---|---|---|
| `<两位编号>` | 00-12 | `03` |
| `<技术域>` | 后端 / 前端 / 静态 / 跨端 / 全栈 | `后端` / `前端` |
| `<资产名>` | 中划线连接，中文为主 | `Controller接口清单` |

### 2.3 技术域变体

> ⚠️ **重要**：`init-asset-docs.sh` 默认产出**变体 2 命名**（`03-后端-Controller接口清单.md` 等含"后端-""前端-"等中缀）。
> 变体 1（单体项目，无技术域后缀）需要 init 后手动重命名。变体 3（多端）需在 09 主端之外，13+ 编号追加。

按项目实际拆分层级命名：

| 原编号 | 变体 1（单体） | 变体 2（前后端分离）✓ **当前默认** | 变体 3（多端） |
|---|---|---|---|
| 03 | `03-API接口清单.md` | `03-后端-Controller接口清单.md` | `03-后端-Controller接口清单.md` |
| 04 | — | `04-后端-Mapper操作清单.md` | `04-后端-Mapper操作清单.md` |
| 05 | — | `05-后端-服务与业务逻辑.md` | `05-后端-服务与业务逻辑.md` |
| 06 | — | `06-后端-安全认证.md` | `06-后端-安全认证.md` |
| 07 | — | `07-前端-页面与组件清单.md` | `07-前端-页面与组件清单.md` |
| 08 | — | `08-前端-状态管理与路由.md` | `08-前端-状态管理与路由.md` |
| 09 | — | `09-静态前台.md` | `09-iOS端.md`、`09-Android端.md`、`09-小程序端.md` |
| 10-12 | 同变体 2 | `10-业务流图.md`、`11-技术债与遗留项.md`、`12-修复建议与优先级.md` | 同 |

**冲突解决**：变体 3 多端时，**主端用 09**，**其它端用 13+** 编号（`13-iOS端.md`）。

**当前 assets/ 默认产出**：
- `docs/assets/` 下的 12 份 `.md.tmpl` 均采用**变体 2 命名**（即"03-后端-Controller接口清单.md"形式）
- 变体 1（单体项目）使用方需自行重命名（`init-asset-docs.sh` 跑完后手动改 03-08 即可）
- 变体 3（多端）需在 init 后额外创建 `13-iOS端.md` 等

**重命名示例**（单体项目）：

```bash
cd asset-docs
for f in 03-后端-*.md 04-后端-*.md 05-后端-*.md 06-后端-*.md \
         07-前端-*.md 08-前端-*.md; do
  newname=$(echo "$f" | sed 's/-后端-/-/; s/-前端-/-/')
  mv "$f" "$newname"
done
```

### 2.4 资产目录（`docs/assets/`）

```
├── 02-er-diagram.png

└── ...
```

## 3. 文档元信息头（必填）

每篇资产**开头**必须包含以下 frontmatter（Markdown 注释块）：

```markdown
<!--
@meta {
  "id": "03-后端-Controller接口清单",
  "version": "1.2.0",
  "last_updated": "2026-06-08",
  "data_source": "实际代码扫描",
  "code_version": "git:abc1234",
  "owner": "Tech Lead @ 团队名",
  "ai_consumable": true,
  "severity_taxonomy": "P0|P1|P2|P3"
}
-->
```

**字段说明**：

| 字段 | 必填 | 说明 |
|---|:-:|---|
| `id` | ✅ | 文件名去后缀 |
| `version` | ✅ | 语义化版本：重大改 = 主版本，小修 = 次版本，勘误 = 修订 |
| `last_updated` | ✅ | YYYY-MM-DD |
| `data_source` | ✅ | `实际代码扫描` / `人工补充` / `AI 抽取 + 人工校验` |
| `code_version` | ✅ | 锚定代码版本的 git hash 或 tag |
| `owner` | ✅ | 维护人/团队 |
| `ai_consumable` | ✅ | AI 编程时是否喂入（`true/false`） |
| `severity_taxonomy` | ❌ 可选 | 本资产使用的严重度体系（`check-meta.sh` 不强制校验） |

## 4. 文档内部结构

### 4.1 标准段落顺序

```
1. 顶部引用块（> 整理时间 · 数据来源）       ← 老 Wxcbrc 模式
2. 目录（> 2000 字时建议有）
3. 概览 / 概览表
4. 详细清单
5. 关键模式 / 代码示例
6. 已知问题（带严重度）
7. 元信息表（文档元信息 + 自动化锚点）
```

### 4.2 严重度等级（统一）

| 等级 | 含义 | 修复 SLA | 标识 |
|---|---|---|---|
| 🔴 P0 | 安全漏洞 / 数据丢失 / 系统不可用 | 24 小时内 | 必修 |
| 🟡 P1 | 功能缺陷 / 体验问题 / 性能 | 1 季度内 | 必排 |
| 🟢 P2 | 架构 / 一致性 / 可维护性 | 半年内 | 计划 |
| ⚪ P3 | 死代码 / 命名 / 注释清理 | 顺便 | 可缓 |

**写入规则**：

- 严禁使用 `P4` 、`P5`
- 跨表/跨文件用同一严重度
- 严重度需在文档开头 frontmatter 声明

### 4.3 表格列规范

**接口清单**的强制列（03）：

| 强制列 | 可选列 | 反例 |
|---|---|---|
| 方法（GET/POST/...） | 公开 | 漏写方法或路径 |
| 路径 | 说明 | 把方法塞进路径 |
| 入参 | 备注 | 路径不写前缀 |
| 鉴权（@PreAuthorize 或公开） | | 入参用"见代码" |
| 严重度（仅 03-12 用） | | 用 emoji 替代说明 |

### 4.4 引用规范

文档内引用其他资产使用**编号锚点**：

```markdown
详见 [02-数据模型与表结构 §3.1](./02-数据模型与表结构.md#3-关键关联关系)
```

引用代码使用**相对路径 + 行号**：

```markdown
`SysUserServiceImpl.save` ([wxcbrc-server/.../SysUserServiceImpl.java:47](file:...))
```

## 5. CHANGELOG.md 规范（**必填**）

> 每次实质性修改必须 bump 修订版本或次版本并写一行。
> "必填"指：合并 PR 时若缺 CHANGELOG 一行，校验脚本会拒绝。

```markdown
# Docs Changelog

## [1.2.0] - 2026-06-15
### 02-数据模型与表结构
- 新增 `linksinfo` 表的 `links_url` 字段重命名说明
### 11-技术债清单
- 新增 P0-05：JWT 密钥硬编码

## [1.1.0] - 2026-05-01
### 03-Controller接口清单
- 补充 UploadFileController 接口
```

**变更粒度**：每次实质性修改（不是错别字）必须 bump 修订版本或次版本。

**CI 校验**：`check-severity.sh` 与 `validate-all.sh` 应同时检查 CHANGELOG 必填（若缺则 WARN）。

## 6. 项目级 AI 引导文件

`CODEBUDDY.md`（或 `CLAUDE.md`、`.cursorrules`）**必须**包含对资产的引用：

```markdown
# 项目名 — AI 引导

> 本项目代码资产位于 `docs/`，请按以下顺序消费：
> 1. [00-文档索引](docs/00-文档索引.md)
> 2. [01-系统总览](docs/01-系统总览.md)
> 3. ...

## 关键约束
- 命名规范：[02-数据模型与表结构 §3.3](./02-数据模型与表结构.md#3-关键关联关系)
- 已知 P0：[11-技术债清单](docs/11-技术债与遗留项.md)
- 修复计划：[12-修复建议](docs/12-修复建议与优先级.md)
```

**反例**：只写一句"参考 docs/"，AI 无法精确消费。

---

## 7. 资产版本控制

- 资产与代码同仓管理
- 资产变更随代码变更走 PR 流程
- `docs/CHANGELOG.md` 单独维护
- **大版本与代码大版本同步**（如代码 v2.0 → 资产 v2.0）

---

## 8. 禁止事项

| ❌ 禁止 | 原因 |
|---|---|
| 把资产拆到多仓管理 | 漂移 |
| 用 Word/PDF 维护资产 | AI 无法消费 |
| 在资产中嵌入大段代码（> 50 行） | 应指向 `file://` 引用 |
| 资产不带元信息头 | 失去可追溯性 |
| 资产不带严重度 | 优先级无法量化 |
| 修改 12 篇编号 | 模板与工具失配 |
| 资产随代码 1:1 复制 | 应"摘要 + 引用" |

---

## 9. 元信息

| 字段 | 值 |
|---|---|
| 强制级别 | 全文强制 |
| 例外处理 | 微库（< 1 万行）可省略 `docs/assets/`，但保留 12 篇 |
| 自动化校验 | 见 06-质量门禁与自检清单 §3 |
