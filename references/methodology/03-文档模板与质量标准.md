# 03 · 12 类文档模板与质量标准（v3.0）

> 每一类资产给出：**目标、必填字段、可选字段、模板示例、质量门禁**。
> 配套可填充模板在 `assets/` 目录。
>
> ⚠️ **v3.0 核心原则**：必须按**代码中实际发生的**进行完整输出，而非按模板套用。

---

## ⚠️ 核心原则：按代码实际输出（最重要）

> **这是资产质量的生命线**。违背此原则，资产将沦为"形式正确但内容空洞"的废料。

### 核心要求

| 原则 | 说明 |
|---|---|
| **不按模板预设清单输出** | 模板的表格/章节是示例骨架，不是必须填充的清单 |
| **按代码实际扫描输出** | 有哪些就列哪些，没有就不写 |
| **从实际出发编号** | 章节编号从实际发现的组件/流/问题出发，不是从模板固定编号出发 |

### 具体示例

| 资产 | ❌ 错误做法 | ✅ 正确做法 |
|---|---|---|
| **10-业务流图** | 按模板预设"主登录/主创建/主查询/权限/上传/改密" | **代码中有哪些业务流就输出哪些**；如果没有"改密流"，就不写 |
| **02-数据模型** | 按模板列"用户表/订单表/商品表" | **从代码扫描 Model/Mapper/XML**，有多少列多少 |
| **11-技术债** | 按模板列"注释代码/命名不一致/死代码/TODO" | **从代码实际扫描发现问题**，没有某类问题就不写那一节 |
| **07-前端组件** | 按模板预设"views/components/Core/components/Business" | **从实际目录结构扫描**，没有 Core 目录就不写通用组件节 |

### 验证方法

每篇资产生成后，必须回答：

| 资产 | 必答问题 |
|---|---|
| 10-业务流图 | "这个流中的每个节点，是否都能在代码中找到对应的实现？" |
| 05-服务逻辑 | "这个业务规则，是否从代码推断而非预设？" |
| 01-系统总览 | "这个架构决策原因，是否有代码证据支持？" |
| 11-技术债 | "这个问题，是否有具体的文件:行号证据？" |

## 00 · 文档索引

**目标**：成为整个 `docs/` 资产的总入口。

**必填字段**：

- 顶部一句话说明本目录是什么、谁来维护
- 表格：12 行（编号、文档、简介）
- 阅读建议：按角色（新人、写新功能、修 Bug、重构）

**模板示例**：

```markdown
# <项目名> 系统存量历史资料 — 文档索引

> 本目录是 **通过读代码** 整理出的存量历史资料。
> 配套项目级规则 `<项目级 AI 引导>` 使用。

## 目录

| 编号 | 文档 | 简介 |
|维度|数量|---|---|
| 01 | [系统总览](./01-系统总览.md) | <一句话> |
| ... | ... | ... |

## 阅读建议

- **接手新人**：先读 01 → 02 → 03 → 07
- **写新功能**：从 02 找数据模型 → 03 找 API 风格 → 05 找业务逻辑
- **修 Bug**：先看 11 已知问题清单
```

**质量门禁**：

- [ ] 12 链接全部可点
- [ ] 阅读建议覆盖至少 3 个角色
- [ ] 不超 200 行

## 01 · 系统总览

**目标**：让任何读者**5 分钟内**了解项目是什么、怎么跑、怎么找。

**必填段落**：

1. **项目背景**（2-3 句：业务定位、目标用户）
2. **项目边界**（目录树 + 跨模块依赖关系）
3. **技术栈**（后端 / 前端 / 静态端 三张表 + 版本）
4. **三端 / 多端关系**（架构图 + 端口 + 通信协议）
5. **关键路径**（按子域列出文件位置）
6. **启动顺序**（Step-by-Step，含验证命令）
7. **数字一览**（代码规模计数：Controller / Service / Model / 端点 / 组件 / 权限串 / TODO）

## 8. 架构决策推理（v3.0）

> 对于每项关键技术选型，追溯「为什么用这个而不是另一个」。

| 技术选型 | 当前方案 | 可能的备选 | 推断原因 | 历史包袱信号 |
|---|---|---|---|---|---|
| ORM 方案 | MyBatis-Plus + XML | JPA | 项目早期 XML，后期 Plus | upms 用注解，devops 用 XML |
| 认证方案 | OAuth2 + JWT | Spring Security | 与老 CAS 兼容 | sso-oauth2-sdk 包名暴露过渡 |
| 注册中心 | Nacos/Consul | Eureka | 支持多种部署 | 三种 runner 变体 |

**推断方法**：
1. 看废弃 import/注释掉的代码
2. 看 `old_*`、`Legacy*` 等旧系统痕迹
3. 看不一致的技术选型

## 9. 未解之谜

| # | 问题 | 重要性 | 推断 |
|---|---|:-:|---|
| 1 | Token 刷新机制的具体实现 | 🔴 | SSOInterceptor 有 isRefreshToken |
| 2 | 账户锁定阈值的触发条件 | 🔴 | 代码中未找到 |

**质量门禁**：

- [ ] 启动顺序每步有可执行命令
- [ ] 关键路径覆盖每个子模块
- [ ] 技术栈版本号齐全
- ***架构决策推理必填***
- [ ] 不超 1000 行

## 10 · 业务流图（端到端）

**目标**：跨层串联，让人理解"一次操作从点击到落库的全链路"。

> ⚠️ **核心原则**：业务流必须**忠实还原代码中的实际调用链路**，不能按模板套用。
>
> 每个节点都要追溯到实际代码位置。流程图是"代码实际发生了什么"的描述，而非"应该发生什么"的模板。

**必填段落**（每个流一节）：

1. **触发条件**（用户从哪一步开始）
2. **完整调用链**（ASCII / Mermaid 流程图，**每个节点必须有代码证据**）
3. **关键决策点**（条件分支、权限校验、白名单比对）
4. **相关代码位置**（文件 + 行号，**必须精确到行**）
5. **已知流相关 Bug**（与 11 对齐）

**必备业务流**（至少覆盖）：

- [ ] 主登录流
- [ ] 主业务创建流（如新闻发布、订单创建）
- [ ] 主业务查询流（如详情页、列表分页）
- [ ] 权限校验流
- [ ] 文件上传 / 导出流
- [ ] 改密 / 找回密码流

**v3.0 深度要求**：

> 每个业务流必须包含：
> 1. **异常路径分析**（≥ 2 个异常场景）
> 2. **决策点识别**（当前决策 + 推断依据 + 合理性评估）
> 3. **隐含假设记录**（代码中未明确的假设条件）

**模板示例**：

```markdown
# 10 · 业务流图（端到端）

> ⚠️ 以下流程基于代码实际追溯，每个节点都有代码证据。

## 1. 后台用户登录

### 正常路径（完整调用链）
```
[用户] 打开 http://localhost:7778/
  → 路由守卫发现未登录 (router/index.js:42)
  → 跳 /login
[Login.vue] mounted (views/login/index.vue:15)
  → refreshCaptcha() (api/captcha.js)
[用户] 提交登录
  → POST /api/user/login (api/user.js:8)
    → ExternalAuthController.login() (Controller:180)
      → HashUtil.decodeBase64() (Utils:42) ⚠️ 密码解密位置
      → LoginAuthentication.checkPassword() (Service:68)
      → Oauth2Service.generateToken() (Service:95)
    → MySQL: SELECT * FROM core_account WHERE username = ? (Mapper:35)
  → 返回 { token, refreshToken }
[前端] 存储 token (store/modules/user.js:25)
  → 跳转首页
```

### 异常路径
| 异常场景 | 触发条件 | 当前行为 | 期望行为 | 影响 | 优先级 |
|维度|数量|---|---|---|---|---|
| 用户名不存在 | 输入未注册用户名 | 抛出 USERNAME_OR_PASSWORD_FAILED | 同左（防枚举） | 无法登录 | 🔴 |
| 账户已锁定 | 错误次数达阈值 | 抛出 ACCOUNT_LOCKED_ERROR | 同左 | 需联系管理员 | 🔴 |

### 决策点
| 决策点 | 当前决策 | 推断依据 | 合理性 | 风险 |
|维度|数量|---|---|:---:|---|---|
| 密码解密位置 | Controller 层 | ExternalAuthController:204 | ⚠️ 违反分层原则 | 安全风险 |
| 错误信息统一 | USERNAME_OR_PASSWORD_FAILED | 不区分原因 | ✅ 防枚举攻击 | — |

### 隐含假设
| 假设 | 推断来源 | 验证方式 |
|维度|数量|---|---|
| 密码 Base64 传输 | HashUtil.decodeBase64() | 抓包验证 |
| 错误计数重置时机 | 未找到重置逻辑 | 登录成功时重置？ |
```

**质量门禁**：

- [ ] **每个节点必须有代码文件:行号证据**（❌ 不能有"→ 调用某个Service"而没有具体位置）
- [ ] 每个流含触发→前端→后端→落库→响应
- [ ] 异常分支有标注
- [ ] 至少 6 个核心流
- [ ] 与 05-业务逻辑编号对齐
- [ ] **自问验证**：这个流中的每个节点，是否都能在代码中找到对应的实现？

## 11 · 技术债与遗留项

**目标**：盘点所有"应该改但还没改"的项目，给出**可分类、可严重度**清单。

> ⚠️ **核心原则**：技术债必须**忠实记录代码中的实际问题**，每条都要有代码证据。

**必填段落**（每个分类一节）：

1. 注释代码块（应清理）
2. 命名不一致
3. 死代码（应删除）
4. TODO / FIXME 清单
5. 硬编码（应外置）
6. 注解 / 标签错位
7. 权限字符串错用
8. 性能问题
9. 版本 / 生态落后
10. 文档与代码不一致

**强制列**：

| 列 | 必填 | 示例 |
|---|:-:|---|
| 严重度 | ✅ | 🔴 P0 |
| 类别 | ✅ | 命名不一致 |
| 位置 | ✅ | `SysUserMapper.xml:45` |
| 问题 | ✅ | `${news_title}` 字符串拼接 |
| 建议 | ✅ | 改 `#{}` 预编译 |

**v3.0 深度要求**：

> 必须包含根因分类和技术债利息估算：

### 根因分类汇总

> 汇总所有技术债的根本原因，用于评估技术债产生模式。

| 根因类型 | 数量 | 典型案例 | 预防措施 |
|维度|数量|---|---|---|
| **复制粘贴** | N | Controller 模板复制后未修改 @Api 标签 | 代码模板审查 |
| **需求变更** | N | 备份文件、遗留 TODO | 变更时清理相关代码 |
| **临时方案** | N | 密码在 Controller 层解密 | 代码审查 |
| **技术选型** | N | MyBatis XML + 注解混用 | 架构评审 |
| **文档缺失** | N | 业务规则未文档化 | 文档即代码 |
| **人员变更** | N | 包名变更未完成 | 交接清单 |

**根因分析说明**：
- **复制粘贴**：代码重复未抽象，改一处忘改另一处
- **需求变更**：需求变了代码没同步更新
- **临时方案**：「先上线再说」留下的 hack
- **技术选型**：选了不合适的技术被迫 workaround
- **文档缺失**：无文档后人看不懂改错
- **人员变更**：交接不清导致知识丢失

### 技术债利息估算

> 估算每类技术债的「利息」——即它对开发效率的影响。

| 根因类型 | 估计影响 | 利息计算 |
|维度|数量|---|---|
| **复制粘贴** | 改 bug 时需改多处 | 每改 1 次 = 实际 3 次 |
| **硬编码** | 改配置需改代码 | 每次发版 +1 小时 |
| **死代码** | 维护时浪费阅读时间 | 每人每天浪费 10 分钟 |
| **版本落后** | 依赖安全漏洞 | 每次升级需 2-3 天 |
| **临时方案** | 逻辑不清晰易出 bug | 新人上手慢 1 周 |
| **文档缺失** | 需逆向推理解读代码 | 每个模块 +4 小时 |

**技术债利息汇总**：
- 每处复制粘贴代码：改 1 处 = 实际 3 处 = +200% 时间
- 每处死代码：每天浪费 10 分钟 = 每年浪费 40 小时

**质量门禁**：

- [ ] 每个 P0/P1 都能追溯到代码位置
- [ ] 不出现"建议重构"等无具体方案
- [ ] 数量 > 20（**深度体检**）
- [ ] 严重度分布合理（P0 < 10%）
- [ ] **根因分类覆盖 6 类**
- [ ] **技术债利息已估算**
