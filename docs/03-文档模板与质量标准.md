# 03 · 12 类文档模板与质量标准

## 00 · 文档索引

**目标**：成为整个 `docs/` 资产的总入口。

**必填字段**：

- 顶部一句话说明本目录是什么、谁来维护
- 表格：12 行（编号、文档、简介）
- 阅读建议：按角色（新人、写新功能、修 Bug、重构）

**模板示例**：

```markdown
# <项目名> 系统存量历史资料 — 文档索引

> 本目录是 **通过读代码** 整理出的存量历史资料。
> 配套项目级规则 `<项目级 AI 引导>` 使用。

## 目录

| 编号 | 文档 | 简介 |
|---|---|---|
| 01 | [系统总览](./01-系统总览.md) | <一句话> |
| ... | ... | ... |

## 阅读建议

- **接手新人**：先读 01 → 02 → 03 → 07
- **写新功能**：从 02 找数据模型 → 03 找 API 风格 → 05 找业务逻辑
- **修 Bug**：先看 11 已知问题清单
```

**质量门禁**：

- [ ] 12 链接全部可点
- [ ] 阅读建议覆盖至少 3 个角色
- [ ] 不超 200 行

## 01 · 系统总览

**目标**：让任何读者**5 分钟内**了解项目是什么、怎么跑、怎么找。

**必填段落**：

1. **项目背景**（2-3 句：业务定位、目标用户）
2. **项目边界**（目录树 + 跨模块依赖关系）
3. **技术栈**（后端 / 前端 / 静态端 三张表 + 版本）
4. **三端 / 多端关系**（架构图 + 端口 + 通信协议）
5. **关键路径**（按子域列出文件位置）
6. **启动顺序**（Step-by-Step，含验证命令）
7. **数字一览**（代码规模计数：Controller / Service / Model / 端点 / 组件 / 权限串 / TODO）

**模板示例**：

```markdown
# 01 · 系统总览

> 整理时间：YYYY-MM-DD · 数据来源：<来源>

## 1. 项目背景
- **系统名**：<名称>
- **业务定位**：<一句话>
- **目标用户**：<角色列表>

## 2. 项目边界
<目录树>

## 3. 技术栈
### 3.1 后端
| 维度 | 技术 | 版本 |
|---|---|---|

## 4. 多端关系
<架构图，ASCII 或 SVG>

## 5. 关键路径
| 用途 | 路径 |
|---|---|

## 6. 启动顺序
1. 启动 MySQL
2. 启动后端
3. 启动前端
4. ...

## 7. 数字一览
| 维度 | 数量 |
|---|---|
```

**质量门禁**：

- [ ] 启动顺序每步有可执行命令
- [ ] 关键路径覆盖每个子模块
- [ ] 技术栈版本号齐全
- [ ] 数字一览与代码实际计数一致（**用 `wc -l`、`grep -c` 验证**）
- [ ] 不超 1000 行

**目标**：让人 / AI **一眼**知道有哪些表、字段含义、关联关系、SQL 模式。

**必填段落**：

1. **实体类与表对应关系**（表：Model、表、字段、备注）
2. **数据库表清单**（表：表名、中文、主要字段、索引）
3. **关键关联关系**（ER 图 + 字段映射约定）
4. **MyBatis XML 命名空间总览**（表：XML、namespace、操作）
5. **关键 SQL 模式**（分页、Join、resultMap、注入风险）

| XML 文件 | namespace | 主要操作 |
|---|---|---|

## 5. 关键 SQL 模式
### 5.1 分页

**质量门禁**：

- [ ] Model 数量 = 实际 `model/*.java` 数量
- [ ] 表数量 = 实际 `*.sql` 数量
- [ ] 每个 Model 有"对应表"
- [ ] ER 图含主外键关系

- [ ] 字段命名约定（小驼峰 vs 蛇形）有说明

## 03 · API 接口清单

**目标**：联调、AI 编程、自动化测试的**事实表**。

**必填段落**：

1. **概览**（表：Controller、前缀、tags、端点数、公开端点）
2. **详细接口清单**（每个 Controller 一节，表格化）
3. **权限字符串全集**（去重表）
4. **公开端点**（无需 token 列表）

**强制列**（每个端点）：

| 列 | 必填 | 示例 |
|---|:-:|---|
| 方法 | ✅ | `GET` / `POST` |
| 路径 | ✅ | `/api/user/findPage` |
| 入参 | ✅ | `PageRequest` / `SysUser` |
| 鉴权 | ✅ | `sys:user:view` / 无 / 公开 |
| 公开 | ❌ | ✅（仅公开端点） |
| 说明 | ✅ | 业务含义 |

**模板示例**：

```markdown
# 03 · 后端 Controller 接口清单

> 共 N 个 Controller。所有路径前缀 `/api`。

## 1. 概览
| Controller | 前缀 | tags | 端点数 | 公开端点 |
|---|---|---|---|---|

## 2. 详细接口清单
### 2.1 UserController（`/api/user`）
| 方法 | 路径 | 入参 | @PreAuthorize | 说明 |
|---|---|---|---|---|

## 3. 权限字符串全集（去重）
| 子系统 | 资源 | view | add | edit | delete | ... |
|---|---|---|---|---|---|---|

## 4. 公开端点（无需 token）
| 端点 | 用途 | 备注 |
|---|---|---|
```

**质量门禁**：

- [ ] 端点总数 = 实际 `@(Get|Post|Put|Delete)Mapping` 数量（用 `grep` 验证）
- [ ] 路径含全局前缀
- [ ] 每个端点有鉴权标注（无 = 需说明）
- [ ] 公开端点单独列
- [ ] 权限串全集与代码 `@PreAuthorize` 一致
- [ ] 不超 1500 行（大型项目可拆 `03a/03b`）

## 10 · 业务流图（端到端）

**目标**：跨层串联，让人理解"一次操作从点击到落库的全链路"。

**必填段落**（每个流一节）：

1. **触发条件**（用户从哪一步开始）
2. **完整调用链**（ASCII / Mermaid 流程图）
3. **关键决策点**（条件分支、权限校验、白名单比对）
4. **相关代码位置**（文件 + 行号）
5. **已知流相关 Bug**（与 11 对齐）

**必备业务流**（至少覆盖）：

- [ ] 主登录流
- [ ] 主业务创建流（如新闻发布、订单创建）
- [ ] 主业务查询流（如详情页、列表分页）
- [ ] 权限校验流
- [ ] 文件上传 / 导出流
- [ ] 改密 / 找回密码流

**模板示例**：

  → 跳 /login

  → refreshCaptcha()
  → ...
```

## 2. 前台免密用户登录
...
```

**质量门禁**：

- [ ] 每个流含触发→前端→后端→落库→响应
- [ ] 异常分支有标注
- [ ] 至少 6 个核心流
- [ ] 与 05-业务逻辑编号对齐

## 11 · 技术债与遗留项

**目标**：盘点所有"应该改但还没改"的项目，给出**可分类、可严重度**清单。

**必填段落**（每个分类一节）：

1. 注释代码块（应清理）
2. 命名不一致
3. 死代码（应删除）
4. TODO / FIXME 清单
5. 硬编码（应外置）
6. 注解 / 标签错位
7. 权限字符串错用
8. 性能问题
9. 版本 / 生态落后
10. 文档与代码不一致

**强制列**：

| 列 | 必填 | 示例 |
|---|:-:|---|
| 严重度 | ✅ | 🔴 P0 |
| 类别 | ✅ | 命名不一致 |
| 位置 | ✅ | `SysUserMapper.xml:45` |
| 问题 | ✅ | `${news_title}` 字符串拼接 |
| 建议 | ✅ | 改 `#{}` 预编译 |

**质量门禁**：

- [ ] 每个 P0/P1 都能追溯到代码位置
- [ ] 不出现"建议重构"等无具体方案
- [ ] 数量 > 20（**深度体检**）
- [ ] 严重度分布合理（P0 < 10%）

## 12 · 修复建议与优先级

**目标**：把 11 转化为**有 SLA、可执行**的修复计划。

**必填段落**（按严重度分节）：

- P0 — 立即修复（24h）
- P1 — 近期修复（1 季度）
- P2 — 计划修复（半年）
- P3 — 顺便清理
- 整体重构路径（远期）

**强制列**：

| 列 | 必填 |
|---|:-:|
| # | ✅ |
| 位置 | ✅ |
| 问题 | ✅ |
| 建议 | ✅ |
| 工作量 | ❌（建议加） |
| 责任人 | ❌ |

**质量门禁**：

- [ ] 每条都能追溯到 11
- [ ] P0 全部有具体修改方案
- [ ] 整体重构路径分阶段
- [ ] 不超 500 行

## 质量门禁：通用 10 条

每篇资产发布前过这 10 条：

1. **元信息头完整**（frontmatter 5 字段）
2. **严重度统一**（P0/P1/P2/P3，无自定义）
3. **可追溯**（位置精确到文件:行号）
4. **可验证**（数字与代码计数一致）
5. **无散文化**（不写散文段落，全部表格化）
6. **交叉引用**（用编号锚点，不用模糊描述）
7. **AI 可消费**（结构化 + 标签 + 元信息）
