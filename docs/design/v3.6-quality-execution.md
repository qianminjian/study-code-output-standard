## 1. 三大硬 Gate（不可跳过的检查点）

### Gate A: 写路径硬约束

**问题**：Worker 把资产写到 `${SKILL_HOME}/asset-docs/`、`/tmp/` 甚至其他模块目录。v3.5 §4.7.2 写了目标路径但无强制。

**修正**：在 SKILL.md Worker Spawn Prompt 的【输出要求】段插入：

```
【输出路径硬性红线（违反 = 输出作废）】
A1. 写入前：在对话中显式声明 "WRITE_TARGET: <绝对路径>"
A2. 写入后：立即 ls -la <该绝对路径> 确认文件存在
A3. 返回摘要含 "PATH_VERIFIED: <实际路径>" 字段
```

编排者收到 Worker 返回后，用 `grep "PATH_VERIFIED:" worker_summary` 提取实际路径，与 `${PROJECT_ROOT}/asset-docs/<NN>-<name>.md` 比对。不一致 → Worker 产出作废，编排者串行补。

**为什么有效**：三阶段验证（写前声明→写后确认→返回比对），任何一步缺失即可判定。

### Gate B: .phase-facts.md 强制校验

**问题**：v3.5 §4.8.1 设计了共享事实字典但无脚本验证。测试中无人检查文件是否完整。

**修正**：新脚本 `scripts/check-phase-facts.sh`（~50 行），验证 5 项：

1. 文件存在 → FATAL 级阻断
2. 文件非空 → FATAL 级阻断
3. 必要段齐全（"Phase 1 完成"、"累积数字一致性"、"跨资产交叉一致性标记"、"模板字段解释示例"）→ FATAL 级阻断
4. Phase ≥ 2 时必须有"跨资产严重度对齐"段 → FATAL 级阻断
5. 数字字段不含空值或 "N/A" 占位 → FATAL 级阻断

**管线集成**：Phase 1→2、2→3、3→4 每次过渡前运行。不通过 → 管线中止，不进入下一 Phase。脚本接入 `check-all.sh` 调用列表。

### Gate C: 跨资产严重度强制对齐

**问题**：Phase 1 的 4 个 Worker 独立判断严重度，同一 `file:line` 在不同资产中标不同等级。v3.5 §7.6"推断发散"预测了但缓解措施未落实。

**修正**：新脚本 `scripts/check-severity-alignment.sh`（~70 行），逻辑：

1. 从 02/03/04/06 四篇资产中提取所有 P0/P1 标记的 `<file>:<line>` 引用
2. 对每个资产对（6 对组合），找共同引用的 `file:line`
3. 检测同一位置在不同资产中的 P 级是否一致
4. 发现不一致 → exit 1，输出冲突清单

**管线集成**：Phase 1 全部 Worker 完成后、Phase 2 启动前运行。不通过 → Phase 1 资产返回修订（标准：取最高严重度），修订后重新检查。Phase 2 启动条件增加"严重度对齐通过"。

## 2. Worker 质量量化：硬指标取代软铁律

v3.5 铁律 4-5（不偷懒、不怕消耗 token）无法脚本验证。v3.6 不假装能验证，用**可脚本量化的指标**取代：

| 原铁律 | v3.6 替代 | 计算方式 | 硬阈值 | 不达标动作 |
|--------|---------|---------|:---:|---------|
| 4. 不偷懒 | **扫描密度** | `grep -c '<file>:'` 实际引用文件数 / `find` 预期目录文件数 | ≥ 0.80 | < 0.80 → 审计 Agent 重点检查，确认偷懒则作废 |
| 5. 不怕消耗 token | **完整度比** | 资产表格实际行数 / `grep` 预期行数 | ≥ 0.95 | < 0.95 → Worker 产出作废，编排者串行补 |
| — | **证据密度** | 含 `<file>:<line>` 的行数 / 总行数 | Extract ≥ 0.30 / Analyze ≥ 0.50 / Synthesize ≥ 0.40 | 不达标标记 `audit_priority: high` |

这三项指标由编排者在 Phase 间检查点计算（集成到 check-phase-facts.sh 的数字段中）。v3.5 §7.8 的设计保持不变，仅将阈值从"建议"升级为"硬通过条件"。

## 4. 多模块质量保障

### 4.1 asset-docs/ 位置硬约束

多模块场景强制单根结构：

```
${PROJECT_ROOT}/asset-docs/
  .batch-progress.json
  .phase-facts.md
  00-CROSS-MODULE.md          # 聚合产出（aggregate-modules.sh 生成）
  <module-a>/
    02-数据模型与表结构.md
    ...
    .module-status.json
  <module-b>/
    ...
```

**pre-flight 检查**（编排者 Step 0 追加）：`find ${PROJECT_ROOT} -maxdepth 3 -name "asset-docs"` 发现非根位置的独立 asset-docs/ → FATAL，拒绝执行。

### 4.2 多模块自动聚合

**新脚本** `scripts/aggregate-modules.sh`（~120 行），Phase 4 全部完成 + 所有模块 `status=complete` 时运行，产出 `00-CROSS-MODULE.md`：

1. 模块清单表（模块名 / 语言 / Controller 数 / 端点 / P0 Bug）— 复用 quality-compare.sh 提取逻辑
2. 跨模块共性问题 — 同一反模式在 ≥ 3 模块中出现 → 架构级债务
3. 技术栈一致性 — 不同模块用不同框架版本 → WARN
4. 全局 P0 排序 — 按严重度 × 影响模块数排名

单模块项目自动跳过，不产出 00-CROSS-MODULE.md。

### 4.3 Agent Stall 自动处置

agent-watchdog.sh 从诊断工具升级为管线必带守护进程：

```
编排者 spawn Phase 1 workers →
  同时 spawn agent-watchdog.sh（每个 worker 输出文件一个 watchdog）
    ↓
watchdog 检测 stall (300s 无输出变化) →
  1. 自动 kill stall agent
  2. 标记 .module-status.json status=failed
  3. 编排者收到 stall event → 该模块降级为串行
  4. 其他模块继续并行（不改 v3.5 全量降级逻辑）
```

**关键改进**：从"全量降级 v3.0"变成"单模块降级串行"。v35-pipeline.sh 在输出"准备就绪"后追加 watchdog 启动指令。
