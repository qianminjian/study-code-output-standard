# study-code-output-standard · 多 Agent 并行架构方案（v3.5）

> 基于 2 个独立设计 Agent（架构视角 + 工人工厂/质量视角）碰撞融合后的统一方案。
> 碰撞日期：2026-06-11
> 第一轮审计：2026-06-11（7 维度审计，2 PASS / 5 有保留 / 0 不通过，2 项修订已应用）
> 第二轮审计：2026-06-11（实施可行性 + 质量健壮性审计，16 项发现已吸收，新增 6 个方案章节）

---

## 0. 为什么需要这个方案

**v3.0 唯一真实痛点**：大项目（>10 万行）单 agent 上下文溢出 → 丢细节、多主题深度研究质量不足。

**之前的设计失败**：
- v3.5.1（3 轮迭代门）：通用 worker prompt 无法覆盖异质资产类型（数据提取 vs 业务流推理 vs 安全分析）
- v4.0（MO+WA+QA 三角色）：15+ 新文件、用 bash 写编排引擎、并行 worker 跨引用矛盾 → 被审计驳回

**本次设计原则**：KISS / YAGNI / 0 新文件（所有逻辑写在 SKILL.md 里）/ 先验证再实施

## 1. 架构总览

### 1.1 角色（无新增 Agent 类型）

| 角色 | 谁 | 职责 |
|------|----|------|
| **编排者** | 调用 skill 的当前 agent | Step 0-2 自己跑；Step 3 按批次 spawn worker；Step 4-7 自己跑（含 01+00 资产） |
| **Worker** | Claude Code `Agent` 工具（`run_in_background: true`） | 读模板 + 读代码 → 写 1 篇资产 .md → 自检 → 返回摘要 |
| **审计 Agent** | Claude Code `Agent` 工具 | Step 4 之后 spawn 1-2 个审计 agent，挑 2 篇资产反向读代码验证 |

### 1.2 核心设计决策（两 Agent 碰撞共识）

1. **编排者 = skill-invoking agent 自身**（不建独立 MO 角色）— 两 Agent 一致同意
2. **文件级隔离，不用 worktree**（worktree 需要 git clean 状态，用户项目无法保证）— Agent A 提出，Agent B 隐含同意
3. **4 种 Worker 认知变体**（按资产认知类型分组）— Agent B 设计，Agent A 认可
4. **4 阶段生成管线**（最大化并行，尊重真实依赖）— Agent A 设计
5. **Worker 直写 asset-docs/**（不经过中间目录）— 两 Agent 一致同意
6. **0 新文件**（worker prompt 逻辑内联到 SKILL.md）— Agent A 设计，Agent B 的 4 个 worker prompt 内容被吸收为内联指令

### 1.3 碰撞分歧与解决

| 分歧点 | Agent A（架构视角） | Agent B（工人工厂视角） | 决议 |
|--------|-------------------|----------------------|------|
| Phase 1 并行数 | 7 worker 全部并行（02-09） | 7 阶段严格串行 | **Agent A 胜出**：02-09 全部读源码，不需要彼此输出。但并发数按模型能力自适应 |
| 01+00 谁写 | 编排者自己写 | Aggregate worker 写 | **Agent A 胜出**：编排者已有全部上下文，不需要再 spawn worker |
| Worker prompt 存储 | 0 新文件，内联到 SKILL.md | 4 个新文件在 workers/ 目录 | **Agent A 胜出**：内联保持 0 新文件，但内容采用 Agent B 的 4 变体结构 |
| 审计 Agent 设计 | 未详细设计 | 详细 2-Agent 审计方案 | **Agent B 补充**：结构审计 + 语义审计 |
| 5 铁律嵌入 | 未明确设计 | 3 层嵌入（preamble + 过程 guard + 自检清单） | **Agent B 补充**：吸收到 worker prompt 内联指令中 |

| 01 系统总览 | 全部资产 | — | **是**（总结） | 编排者 |
| 00 文档索引 | 全部资产 | — | **是** | 编排者 |

### 2.2 4 阶段管线

```
Phase 1 ─── 7 worker 并行（无依赖，全部读源码）
│  W-02  W-03  W-04  W-06  W-07  W-08  W-09
│  全部 run_in_background: true，同时启动
│  ⚠️ 实际 worker 数取决于项目类型：全栈 7 个，纯后端 4 个（跳过 07/08/09），
│     纯前端 3 个（跳过 02/03/04/05/06，详见 §4.7.3 映射表）
│  编排者等待全部完成（或容忍失败）
▼
Phase 2 ─── 1 worker（可选依赖 Phase 1 的 03+04 输出）
│  W-05 服务逻辑
│  注：05 也能独立读 ServiceImpl，但对照 03+04 产出可提升质量
▼
Phase 3 ─── 1 worker（依赖 Phase 1+2 全部输出）
│  W-10 业务流图 ← 跨层调用链追踪
▼
Phase 4 ─── 并行启动 11+13，然后 12
│  W-11 技术债 + W-13 反模式（并行）
│  等待 W-11 完成 →
│  W-12 修复建议（派生自 W-11）
▼
编排者 ── 01 系统总览 → 00 文档索引
```

### 2.3 Phase 1 并行可行性分析（v3.5 审计修订）

Agent B 的 7 阶段串行设计过于保守。逐个验证 02-09 资产的源码读取范围：

| Worker | 读什么 | 与其他 Worker 的文件重叠 | 输出依赖？ | 可并行？ |
|--------|--------|------------------------|-----------|----------|
| W-02 数据模型 | `model/` + MyBatis XML | 与 W-04 **共享 XML 目录** | 否 | ✓ |
| W-03 API 清单 | `controller/` | 与 W-06 **共享 Controller 文件**（06 需读 `@PreAuthorize`） | 否 | ✓ |
| W-04 Mapper 清单 | `mapper/` + MyBatis XML | 与 W-02 共享 XML，与 W-06 **共享 XML**（OWASP SQL 注入扫描） | 否 | ✓ |
| W-06 安全认证 | SecurityConfig + Filter + **跨目录**（Controller 注解 + XML SQL 检查） | **与 W-02/03/04 均有重叠** | 否 | ✓ |
| W-07 前端页面 | Vue/React 组件 | 与 W-08 边界模糊（router 文件可能共享） | 否 | ✓ |
| W-08 前端状态 | Store + Router | 与 W-07 边界模糊 | 否 | ✓ |
| W-09 静态站 | 静态 HTML | 无重叠 | 否 | ✓ |

**关键澄清**：存在**文件读取重叠**（特别是 06 与 02/03/04 的跨目录扫描），但**不存在输出依赖**——7 个 worker 各自读到同一文件的不同方面，独立产出结论，最终写到不同的 `asset-docs/<NN>-*.md`。

- W-02 从 XML 提取 resultMap 和数据关系
- W-04 从同一 XML 提取 SQL 语句和 mapper 操作
- W-06 从同一 XML 扫描 SQL 注入模式

三者读的是同一批文件的不同维度，互不冲突。交叉引用的一致性（如 entity 名在 02 和 04 中是否一致）留到 Step 4 校验阶段统一处理——v3.0 串行生成时，同一个 agent 在写 04 时已经读过 02 的产出所以天然一致；并行时这个一致性被推迟到校验阶段，但校验量没有增加（v3.0 也需要 human review 抽查）。

## 3. 动态自适应并发

### 3.1 模型能力检测

编排者在 Step 0 检测自身模型：

| 检测方式 | 信号 |
|---------|------|
| 环境变量 `CLAUDE_CODE_MODEL` | `opus` / `sonnet` / `haiku` |
| 系统提示中的模型标识 | 匹配 `claude-opus` / `claude-sonnet` / `claude-haiku` |
| 当前上下文剩余 token 估算 | > 150K = 高，100-150K = 中，< 100K = 低 |

### 3.2 并发决策表

| 模型 | Phase 1 并发数 | Worker 分配策略 |
|------|:-----------:|----------------|
| **Opus（高能力）** | 5-7 并行 | 每个资产独立 worker，全部 `run_in_background: true` 同时启动。Worker 是独立 Agent 调用（各自维护文件读取上下文），无法共享文件描述符，但文件读重叠可以接受——见 §2.3 分析 |
| **Sonnet（中能力）** | 3-4 并行 | Worker 1: 02+04（共享 XML 读取，两篇资产的变体指令合并为一个 prompt，worker 内部串行产出两篇 .md），Worker 2: 03+06（共享 Controller 文件，同内部串行），Worker 3: 07+08+09（前端全部，内部串行）。合并读取重叠文件的资产到同一 worker，减少重复 I/O。**注意**：合并 worker 需在 prompt 中明确区分各资产的变体类型和独立自检清单，避免模板混淆 |
| **Haiku / 低 context** | 0 并行 | **降级到 v3.0 串行**（Haiku context 太紧，多 agent 开销 > 收益） |

### 3.3 动态降级触发

编排者在以下情况自动降级并发：
- Step 1+2 勘探/分类消耗 > 50K tokens → 降一档（Opus 7→4, Sonnet 4→2）
- Agent 工具 spawn 失败 2 次 → 降级到 v3.0 串行
- 项目 < 1 万行 → 直接走 v3.0 串行（多 agent 开销不值得）

## 4. Worker 认知变体（4 种）

> 内容来自 Agent B 的详细设计，吸收为 SKILL.md 内联指令（不建独立文件）

### 4.1 变体矩阵

| 变体 | 适用资产 | 核心认知任务 | 关键 guard |
|------|---------|-------------|-----------|
| **A: Extract** | 02, 03, 04, 07, 08, 09 | grep→提取→填表→计数校验 | grep 数 = 表格行数 |
| **B: Analyze** | 06, 11, 13 | 模式匹配→严重度分类→根因分析→证据引用 | 每条结论带 `<file>:<line>` |
| **C: Synthesize** | 05, 10 | 调用链追踪→语义重建→决策点识别→异常分支 | 如实报告异常路径/决策点/隐式假设数，不足基线时解释 + @incomplete-coverage |
| **D: Aggregate** | 01, 00, 12 | 读上游资产→交叉校验→综合→排序 | 不复制粘贴，必须合成 |

### 4.2 5 铁律嵌入（3 层）+ 可验证性分级

Agent B 设计的 3 层嵌入策略被采纳：

**Layer 1 — Worker Prompt 共享 Preamble**：
```
5 铁律（违规 = 输出作废）：
1. 完全反应：代码中每个实体/端点/类都必须出现在输出中
2. 不丢失：grep 计数 = 表格行数，差 1 就重新扫
3. 不幻觉：每条结论带 <file>:<line> 证据，找不到写"未确认"
4. 不偷懒：深度分析是强制的，不抽样，不全读等于没读
5. 不怕消耗token：宁可读 50 个文件产出完整资产，不要读 10 个文件产出浅薄资产
```

**Layer 2 — 过程级 Guard**：每个变体的扫描流程中嵌入铁律检查点（见 §4.3-4.6）

**Layer 3 — 自检清单**：每个 worker 提交前必须过 5-8 项自检清单

**可验证性分级**（v3.5 审计修订 · 诚实标注）：

| 铁律 | 自动化验证 | 审计 Agent 验证 | 人类可验证 | 诚实度 |
|------|:---:|:---:|:---:|------|
| 1. 完全反应 | **高**（grep 计数 vs 表格行数，check-all.sh 可测） | **高**（抽样 Read 源文件，确认遗漏） | 高 | **硬铁律** |
| 2. 不丢失 | **高**（同铁律 1，差异 ≥ 1 即为失败） | 中（抽样验证，无法全覆盖） | 高 | **硬铁律** |
| 3. 不幻觉 | **中**（`<file>:<line>` 格式可脚本校验，但内容正确性需语义判断） | **高**（审计 Agent 逐条 Read 验证） | 高 | **硬铁律** |
| 4. 不偷懒 | **低**（无法自动检测 worker 是否读了 10 个文件还是 50 个） | **中**（审计 Agent 可通过"缺失发现"间接推断，但无法直接验证读取量） | 低 | **软铁律**（依赖 worker 自觉 + audit 间接推断） |
| 5. 不怕消耗 token | **无**（无法测量、无法验证） | 无 | 无 | **心态声明**（不是铁律，是"宁可过度不可不足"的设计理念） |

> **诚实结论**：铁律 4 和 5 不是通过工具/脚本强制执行的"硬铁律"，而是通过 worker prompt 的行为指令 + 审计 Agent 的间接推断来软约束。承认这一局限比假装"5 条都可验证"更诚实。v3.0 串行模式同样面临这个问题（单 agent 也可能偷懒），v3.5 的改进是通过**强制最小量**（如 Synthesize 的 ≥2 异常路径）和**审计 Agent 的抽查**来增加偷懒的代价。

### 4.3 变体 A: Extract（数据提取）

```
=== 扫描流程 ===
1. 运行 grep/find 命令，记录精确计数
2. 对每个找到的文件 Read 全文，不跳过
3. 按模板结构提取数据到表格
4. 填完所有表格后重新 grep 计数，行数必须一致

=== 强制校验 ===
[ ] grep entity 数 = 表格行数
[ ] 每行有源码引用（<file>:<line>）
[ ] 无 DDL 时启用 §2-B 推断模式 + ⚠️ 标记
[ ] @meta frontmatter 字段填齐
[ ] 模板末尾自问验证题已回答
```

### 4.4 变体 B: Analyze（模式分析）

```
=== 分析流程 ===
1. 跑 scan-antipatterns.sh
2. 对每个命中 Read 源文件，确认不是误报
3. 手动 grep 补充脚本遗漏的模式（TODO/FIXME/password/secret）
4. 对 06 安全：逐项对照 OWASP Top 10
5. 每条发现分类：严重度 + 根因类型（复制粘贴/需求变更/临时方案/技术选型/文档缺失/人员变更）

=== 强制校验 ===
[ ] 每条发现带 <file>:<line> 证据
[ ] P0 < 10% 总数
[ ] scan-antipatterns.sh 输出已消化
[ ] 手动 grep 补充扫描已执行
[ ] 根因分类完成
[ ] 06: OWASP 10 项逐项回应（不适用也写明）
[ ] 未解之谜 ≥ 1 条
```

### 4.5 变体 C: Synthesize（跨文件推理）

```
=== 推理流程 ===
Phase 1: 入口发现 — grep 所有 Controller 入口点
Phase 2: 调用链追踪 — 对每个入口点追踪完整链路
         Controller.method() → Service.method() → Mapper.method() → SQL
         每个节点 Read 实际源码，记录：文件位置、方法签名、关键逻辑、错误处理
Phase 3: 语义重建 — 对每条链路重建：
         - 正常路径、决策点（每个 if/switch）、异常路径（每个 catch/throw）
         - 隐式假设（硬编码值、magic numbers）
         - 业务规则（从校验注解、正则、常量推断）

=== 强制最小量（反偷懒门 · v3.5 第二轮审计修订） ===
⚠️ 以下为"基线期望"而非硬阈值。worker 必须如实报告实际发现数，
若低于基线必须解释原因并标记 @incomplete-coverage（供审计 Agent 重点检查）。

[ ] 每条业务流：记录实际异常路径数（catch/throw），若 < 2 条 → 解释"代码中确实少于 2 处"或"文件读取受限，以下路径未追踪: ..."
[ ] 每条业务流：记录实际决策点（if/switch），若 < 1 个 → 解释原因
[ ] 每条业务流：记录实际隐式假设（硬编码值/magic numbers），若为 0 → 解释"经全文搜索未发现"
[ ] 未解之谜 ≥ 3 条（不强制，但少于此数需说明"以下区域已确认无未解之谜: ..."）
[ ] 每个调用链节点追踪到实际代码（<file>:<line>）

=== 修订理由 ===
第二轮审计发现：硬阈值（"每条业务流 ≥ 2 个异常路径"）可能诱导 worker
在没有足够异常路径时虚构发现。改为"如实报告 + 不足时解释 + 标记 @incomplete-coverage"，
让审计 Agent 能识别"真正简单"vs"偷懒跳过"的区别。
```

### 4.6 变体 D: Aggregate（综合汇总）

```
=== 汇总流程 ===
1. 通读全部上游资产（不是摘要，是全文）
2. 提取关键数据点
3. 交叉一致性检查（entity 名在 02/03/05/10 中一致、API 端点 03=07 前端调用匹配）
4. 综合，不是复制粘贴

=== 强制校验 ===
[ ] 全部上游资产已读全文
[ ] 交叉一致性检查完成，不一致处已标出
[ ] 01: 含架构决策推论 + 历史包袱信号
[ ] 12: 每条 P0 from 11 都有对应修复方案 + 工作量估算 + SLA
[ ] 00: 含角色化阅读路径（新人/修bug/架构师）
```

### 4.7 编排者 Prompt 组装逻辑（第二轮审计补充 · 实施可行性 Agent 发现）

> 这是第一轮方案最大的缺失：只定义了 worker 该做什么，没定义编排者**如何**把 SKILL.md 内联指令翻译成 Agent 工具调用。

**4.7.1 变量来源解析**

编排者在组装 prompt 前必须解析以下变量：

| 变量 | 来源 | 解析方式 |
|------|------|---------|
| `${PROJECT_ROOT}` | Step 0 用户确认 | `pwd` 或 `--path` 参数 |
| `${SKILL_HOME}` | 当前 skill 安装位置 | SKILL.md 所在的 repo root（`${SKILL_HOME}` 由 skill 系统注入） |
| `${SRC_DIR}` | Step 1 勘探 | 从项目结构推断（`src/main/java` 或 `src/` 或 `.`） |
| `${WEB_SRC}` | Step 1 勘探 | 前端源码目录（`src/web/` 或 `src/main/resources/static/`） |
| `${ASSET_DIR}` | `${PROJECT_ROOT}/asset-docs` | 固定拼接 |
| `${MODEL}` | 环境变量 / 系统提示 | `CLAUDE_CODE_MODEL` 或模型标识匹配 |
| `${TEMPLATE_PATH}` | `${SKILL_HOME}/assets/<NN>-*.md.tmpl` | 固定拼接 + 资产编号映射 |

**4.7.2 Worker Spawn 通用模板**

编排者调用 `Agent` 工具时，按以下结构组装 prompt（伪代码）：

**4.8.2 Context 检查点**

在启动 Phase 3（W-10，认知最重的 worker）前，编排者必须检查自身 context 剩余量。若 < 30K → 输出 checkpoint 摘要 + 提示用户"上下文紧张，建议新开会话继续，Phase 1+2 资产已保留在 asset-docs/ 中"。

**4.8.3 降级检查点**

| 检查点 | 条件 | 动作 |
|--------|------|------|
| Phase 1 → 2 | ≥ 3 个 worker 失败 | 清理 Phase 1 产出，降级 v3.0 串行 |
| Phase 1 → 2 | 编排者 context < 50K | 输出已完成的 Phase 1 资产，建议用户新开会话 |

## 5. 审计 Agent 设计

> 来自 Agent B 的详细设计

### 5.1 结构审计（自动）

在 Phase 4 之后、编排者写 01+00 之前运行：

1. `bash check-all.sh`（已有 4 子脚本）
2. 额外多 worker 一致性检查：
   - 02 entity 名在 03/05/10 中出现时拼写一致
   - 12 修复 ID 全部映射到 11 发现位置
   - 01 数字一览与实际各资产计数一致

### 5.2 语义审计（AI Agent）

Spawn 1 个审计 Agent，挑 **3 篇资产**反向读代码验证：
- **必选 1 篇 Synthesize 资产**（05 或 10）— 幻觉风险最高
- **必选 1 篇 Analyze 资产**（06 或 11）— 漏扫风险最高
- **随机选 1 篇 Extract 资产**（02/03/04/07/08/09 任一）— 作为 Phase 1 输入基础的代表性抽查。Extract 资产是后续 Phase 2-4 的输入基础，若 Extract 有系统性问题，整个管线受影响（garbage in, garbage out）

审计流程：
1. 读目标资产全文，提取所有 `<file>:<line>` 引用
2. 挑 5 条声明验证（2 条主表数据 + 1 条深度分析 + 1 条问题发现 + 1 条数字声明）
3. 逐条 Read 引用位置的源码，对比声明 vs 实际代码
4. 评分：PASS（完全一致）/ MINOR（位置偏了但内容对）/ MAJOR（部分错误）/ FAIL（与代码矛盾）
5. 输出审计报告

审计门禁：
- 0 FAIL → 通过
- 1-2 MAJOR → 该篇资产返回对应 worker 修订，修订后重新审计
- ≥1 FAIL → 该 worker 类型全部输出拒绝，编排者串行补该类型所有资产
- ≥3 MINOR → **资产返回 worker 修订**（v3.5 审计修订：之前版本为"接受但标记"，过于宽松。3 个 MINOR 同篇资产说明 worker 不够仔细，应要求修订）

### 5.3 审计可靠性增强（第二轮审计补充 · 质量健壮性 Agent 发现）

**5.3.1 审计 Agent 学科 Prompt**

审计 Agent 本身也需要约束。编排者 spawn 审计 Agent 时，prompt 头部必须包含：

```
=== 审计学科（违规 = 审计结果作废） ===
1. 你只能验证，不能重写资产。如果发现错误，标记它，不要修复它
2. 每条评分必须有 READ 证据。你说"与代码矛盾"时，必须引用具体源码片段
3. 不确定时标 UNCERTAIN，不要猜测。UNCERTAIN 不扣分
4. 你没有发现错误 ≠ 资产正确。你只证明了"这 5 条抽样的这 5 个位置没错"
5. 不要因为"看起来合理"就给 PASS。PASS 要求源码逐字对照一致
```

**5.3.2 争议升级机制**

当 Worker 与审计 Agent 对同一发现评分不一致时（如 Worker 认为 PASS，审计 Agent 评为 MAJOR）：

| 场景 | 处理 |
|------|------|
| 1 个审计 Agent MAJOR → 1 个 Worker 不服 | 编排者自己读源码裁决（不 spawn 第三方） |
| 审计 Agent FAIL → Worker 不服 | 编排者读源码裁决，FAIL 成立则 Worker 该篇重做 |
| 两个审计 Agent 对同一发现评分矛盾 | 编排者读源码裁决，取更保守的评分（更接近 FAIL 的那个） |

**5.3.3 审计覆盖率诚实声明**

当前审计覆盖率 3/14 = 21%。方案明确承认：
- 审计 Agent 采样的是"**高风险样本**"（Synthesize + Analyze + 随机 Extract），不是随机样本
- 21% 覆盖率不能统计学推广到未审计的 79%
- 未被审计的资产标记 `audit_status: unaudited`，供人类 reviewer 优先检查
- 审计的价值不在于覆盖，而在于**早期信号**——如果 3 篇高风险资产中有 2 篇 FAIL，说明整个 Phase 可能有问题

**5.3.4 审计 Agent 间信度**

当同时 spawn 2 个审计 Agent（可选增强模式，仅 Opus 上下文充足时启用）：
- 两个 Agent 独立审计同一篇 Synthesize 资产（幻觉风险最高）
- 交叉对比两者的 FAIL/MAJOR 发现：一致发现 → 高置信度；不一致发现 → 编排者手动裁决
- 若两个 Agent 均未发现 FAIL → 资产通过审计门禁

## 6. Wall Time 测量协议

> 来自 Agent A 的详细设计

### 6.1 测试项目

**wxcbrc**（~1.5 万行，全栈 Java + Vue + 静态 jQuery）— 方法论中反复引用的标准参考项目。全部 12 种资产类型适用。

### 6.2 测量方法

1. **冻结基线**：git checkout 固定 commit hash，确保两次跑的代码完全一致
2. **v3.0 串行基线**：跑 3 次，记录 wall time（从 skill 触发到 Step 7 完成），计算 mean ± std
3. **v3.5 并行**：跑 3 次，记录 wall time，计算 mean ± std
4. **采集指标**：
   - 总 wall time
   - Phase 1 worker spawn→全部完成时间
   - 单 worker 耗时（最长/最短/平均）
   - check-all.sh 通过率
   - 人工 review 3 篇资产（02/03/10）完整性

### 6.3 通过标准

- 并行 wall time **不快于**串行 → 不通过（并行开销 > 收益）
- check-all.sh 通过率差异 > 5% → 不通过
- 人工 review 发现资产完整度退化 → 不通过

## 7. 容错与降级

### 7.1 Worker 失败处理

| 场景 | 处理 |
|------|------|
| Worker spawn 失败（Agent 工具报错） | 重试 1 次，仍失败 → 编排者自己串行生成该篇 |
| Worker 返回但自检不通过 | 编排者发回修订指令（带具体问题），重跑 1 次 |
| Worker 产出文件存在但完成度指标不达标（§7.8） | 编排者发回补充指令（带不达标指标），重跑 1 次 |
| Worker 产出文件不存在或为空 | 编排者自己串行生成该篇 |
| 审计 Agent 发现 FAIL | 该 worker 类型全部输出拒绝，编排者串行补该类型所有资产 |
| Phase 1 超过 3 个 worker 失败 | **全量降级 v3.0 串行**（多 agent 不可靠） |

**渐进退化检测**（第二轮审计补充）：当前方案用"全或无"判断 worker 成败，但更危险的是**部分成功**——文件写出来了，@meta 校验通过，但内容漏了 30%。检测方式：
- 完整度比 < 0.80（预期 50 个端点，实际只写了 35 个）→ 返回修订
- 两个独立 worker 对同一事实有不同的数字声明（如 W-03 说 67 个端点而 W-06 说 72 个）→ 编排者 grep 裁决
- **孤立端点检测**：Phase 1 完成后，编排者 grep 检查是否有 Controller 端点未被任何资产引用（在 W-03/06 中都缺失）→ 发现孤立端点 → 两篇资产都返回补充

### 7.2 全量降级条件

以下任一条成立 → 退出多 agent 路径，走 v3.0 串行：
1. Go/No-Go 门不通过
2. Phase 1 超过 3/7 worker 失败
3. 编排者 context 剩余 < 20K tokens（Step 1+2 后）
4. 项目 < 1 万行
5. 用户显式指定 `--mode=v3`

降级时清理所有 Phase 1 部分产出（可选：保留已完成的部分供用户参考）

**降级风险**：如果并行方案本身不稳定（worker 失败率 > 43% = 3/7），用户的实际体验是先等 Phase 1 所有 worker 跑完（最长 worker 耗时），发现失败 ≥ 3 个，再跑一遍串行（完整 v3.0 耗时）= 总计 **并行耗时 + 串行耗时**，比直接跑串行更慢。缓解措施：Gate 1 的 min-test 先验证单 worker 稳定性；Phase 1 中第 3 个 worker 失败时立即触发降级（不等剩余 worker 完成）。

### 7.3 Worker 完成检测机制

编排者通过以下方式判断 worker 是否完成：

1. **Agent 工具返回值**：`run_in_background: true` 的 Agent 工具在 worker 完成后返回摘要。编排者等待所有 Phase 1 worker 的 Agent 工具调用返回（Claude Code 中 Agent 工具在 background 模式下仍然会有返回值，只是不阻塞编排者的其他操作）
2. **文件存在性检查**：作为兜底，编排者在收集返回值后检查 `asset-docs/<NN>-*.md` 是否存在且非空
3. **内容有效性检查**：`head -3 asset-docs/<NN>-*.md | grep "@meta"` — 快速验证文件是合法资产而非骨架占位
4. **超时处理**：如果某个 worker 超过 15 分钟未返回，编排者视为失败，触发重试或串行补

### 7.4 生成管线瓶颈分析

并行加速比受以下瓶颈约束：

| 瓶颈 | 位置 | 影响 | 缓解 |
|------|------|------|------|
| **最慢 worker** | Phase 1 | 总耗时 = max(各 worker 耗时)，而非 sum。W-02（100+ entity）可能比 W-09（静态 HTML）多 3-5x 耗时 | 模型自适应：Opus 分配慢 worker，Sonnet 分配快 worker。实际加速比取决于 worker 耗时的方差 |
| **串行 Phase 3** | W-10 业务流图 | 必须等 Phase 1+2 全部完成才能开始。W-10 是认知负担最重的任务（跨层调用链追踪），是整个管线的**单点瓶颈** | W-10 分配给能力最强的模型（Opus）。Phase 1+2 完成后立即 spawn，不排队 |
| **串行编排者** | 01+00 | 编排者需通读全部 12 篇资产后才写 01+00。此步骤在 v3.0 中也存在，不是并行引入的新瓶颈 | 无额外缓解，但也不是新问题 |

加速比预估（Phase 1 方差中等情况下）：
- Phase 1：2-4x（7 workers 并行，瓶颈 = 最慢 worker）
- 整体管线：1.5-2.5x（Phase 3 + 编排者仍是串行）
- 大项目 context 溢出场景：**质量提升是主要收益，wall time 加速是次要收益**

### 7.5 已知局限与接受的风险

| 局限 | 为什么接受 | 谁做的决定 |
|------|----------|-----------|
| Agent 工具 `background: true` 行为未经实测 | Gate 1 专为此设计。不通过则方案终止。这是最危险的假设，因此放在第一个 Gate | 两 Agent 共识：必须测，不测不实施 |
| 5 铁律中 2 条无法硬验证 | v3.0 同样无法验证。v3.5 的改进是通过强制最小量 + 审计抽查增加软约束，而非假装能硬验证 | Agent B 设计 + 审计修订 §4.2 诚实标注 |
| 并行可能比串行更慢（worker 方差大 + 失败重跑） | Gate 2 专为此设计。不通过则方案终止。wxcbrc 小项目上可能加速比 < 1，但大项目上质量收益仍然存在 | Agent A 设计：测了再决定 |
| SKILL.md 膨胀到 ~756 行 | 0 新文件的代价。比 v4.0 的 15+ 新文件好。如果未来超 800 行再考虑拆文件（YAGNI——现在不需要） | Agent A 设计 |
| 审计覆盖率 21%（3/14）| 比 v3.0 的 0% 审计覆盖率好。全量审计需要人工，21% 的 AI 审计 + 人类 review 是务实平衡 | Agent B 设计 + 审计修订 §5.2 |

### 7.6 并行特有缺陷模式（第二轮审计补充 · 质量健壮性 Agent 发现）

并行生成引入了 v3.0 串行模式不存在的 4 种缺陷模式。方案必须在 worker prompt 和校验阶段显式防御：

| 缺陷模式 | 描述 | 发生条件 | 影响资产 | 缓解措施 |
|---------|------|---------|---------|---------|
| **命名空间推断冲突** | W-02 称某类为 `UserEntity`，W-03 在 Controller 中看到 `User` 引用，自行推断为不同 entity（实际是同一 entity 的简写） | Entity 类名在 Controller/Service 中以简写形式出现 | 05, 10（跨资产引用时产生虚假的"不一致"） | Phase 1 共享事实字典收录 W-02 的 entity 全限定名列表；W-03/05/10 prompt 指示"entity 名以 W-02 产出为准" |
| **推断发散** | 两个 worker 从同一段代码推断出不同的业务含义。例如 W-05 从 `if (status == 3)` 推断"状态 3 = 已取消"，W-10 从同一段代码推断"状态 3 = 已退款" | 代码中有 magic number 或缺乏注释的状态常量 | 05, 10（业务逻辑描述自相矛盾） | W-10 prompt 要求"对照 W-05 的业务规则描述，若发现不一致标出，不确定时标 UNCERTAIN"；check-consistency.sh 校验 05 与 10 的状态值定义交集 |
| **引用衰减** | W-10 读 W-02 资产时看到 entity 列表，但不知道每个 entity 的完整字段语义（v3.0 串行时同一个 agent 写 W-02 时有完整上下文）。W-10 引用 entity 名但丢失了字段级的精确性 | 跨 Phase 依赖（Phase 3 读 Phase 1 产出） | 10（业务流图中 entity 字段引用不精确） | W-10 prompt 指示"对每个引用的 entity，Read 源 Entity 类确认字段名（不要只依赖 W-02 产出的表格摘要）" |
| **模板解释漂移** | 不同 worker 对同一模板字段的理解不一致。例如 W-02 在"说明"列写业务含义，W-04 在"说明"列写技术实现细节 | 模板字段语义不够精确（如"说明"列含义模糊） | 交叉引用一致性（人读起来感觉"风格不统一"） | 共享事实字典附带"模板字段解释示例"（从编排者 Step 1 分析中提取 1-2 个正例）；Sonnet 合并模式下编排者在 prompt 中显式给出字段填写示例 |

### 7.7 质量机制对比：v3.0 → v3.5（第二轮审计补充 · 质量健壮性 Agent 发现）

v3.0 串行模式有 5 项隐性质量机制。转移到并行模式后，部分被削弱，需要用显式机制替代：

| v3.0 机制 | 在 v3.5 中的状态 | 退化风险 | v3.5 替代机制 |
|-----------|----------------|---------|-------------|
| **串行依赖排序**（01 最后写，天然有全局视角） | **保留**：Phase 管线遵守同一依赖顺序 | 无退化 | 管线设计（§2.2） |
| **共享上下文一致性**（同一 agent 写 02 时读过的 entity，写 04 时还记得） | **削弱**：并行 worker 没有共享记忆 | W-02 与 W-04 对同一 entity 拼写不一致 | 共享事实字典（§4.8.1）+ check-consistency.sh 交叉校验 |
| **人类抽查**（Step 4 人类抽 5-10 端点 + 3-5 业务流） | **保留但需加强**：并行产出的交叉不一致风险更高，抽查量应增加 | 并行可能引入新类型错误（如命名空间冲突） | 人类抽查量从 5-10 → 8-12（增加 50%）；优先抽查 Phase 1 有文件读取重叠的资产（02/03/04/06） |
| **同行校验**（另一人过一遍） | **不变** | 无退化 | 同行校验 |
| **业务流人工走读**（Step 6.5 必校验 1-2 核心流） | **保留但更关键**：W-10 从冷启动追踪调用链，比串行时同一 agent 的记忆更不可靠 | W-10 证据链断裂风险更高 | W-10 走读从 1-2 流 → 2-3 流；优先走读最长/最复杂的调用链 |
| **（v3.0 没有）** | **新增**：AI 审计 Agent 抽样验证 | — | 语义审计（§5.2）+ 审计可靠性增强（§5.3） |

**质量等价性论证（诚实修订）**：

第一轮方案声称"并行产物质量等价或更好"。第二轮审计指出这个论证逻辑断裂：21% 审计覆盖率不能证明 79% 未审计资产的质量。

修订后的论证：
- v3.5 **不声称**并行产物质量 ≥ 串行产物质量（无证据）
- v3.5 **声称**：通过"强制最小量 + 共享事实字典 + 审计 Agent 抽样 + 人类抽查增量"，v3.5 能**检测到**大部分质量退化，并在发现退化时触发修订
- 最终质量取决于修订循环的执行力，而非单次生成的完美度
- Gate 3 是最终裁决者：如果并行版主观弱于串行版，方案放弃

### 7.8 Worker 完成度指标（第二轮审计补充 · 质量健壮性 Agent 发现）

当前方案仅用"文件存在 + @meta 校验"判断 worker 完成。这是必要的但不充分——文件可以存在但内容空洞。新增以下 3 项完成度指标：

| 指标 | 计算方式 | 阈值 | 不达标动作 |
|------|---------|------|-----------|
| **完整度比** | 实际表格行数 / grep 预期行数 | ≥ 0.95 | < 0.95 → 返回 worker 补充 |
| **证据密度** | 含 `<file>:<line>` 引用的行数 / 总行数 | Phase 1 Extract: ≥ 0.30; Analyze: ≥ 0.50; Synthesize: ≥ 0.40 | 低于阈值 → 审计 Agent 重点检查该篇 |
| **自检通过率** | 变体自检清单通过项 / 总项 | = 1.0 | < 1.0 → 返回 worker 修订（提供未通过项清单） |

编排者在 Phase 间检查点验证这些指标，不达标的资产不入下一 Phase 依赖。

## 9. Go/No-Go 验证门（实施前必过）

### Gate 1: Agent 工具可行性

**Gate 1.0 — 参数名验证**（第二轮审计补充 · 实施可行性 Agent 发现）：

**Gate 1.1 — 最小端到端测试**：
1. 创建临时目录含 1 个 Controller 类

**通过**：agent 能读代码→写文件→返回摘要  
**失败**：Agent 工具不可用，或 worker 不能写文件，或 worker 全量输出回到编排者 context → 放弃多 agent，v3.0 串行就是答案

按 §6 协议在 wxcbrc 上跑 3 次对比。

**通过**：并行 wall time ≤ 串行 wall time  
**失败**：多 agent 开销吃掉并行收益 → 放弃

### Gate 3: 资产质量不退化

人工对比 3 篇资产（02/03/10）的并行版 vs 串行版。

**通过**：主观等价或更好  
**失败**：并行版丢细节 → worker prompt 需修订后再测
