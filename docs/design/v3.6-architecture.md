# study-code-output-standard · v3.6 设计方案

> 设计日期：2026-06-12
> 设计基准：v3.5-parallel-architecture.md（788 行第二轮审计版）
> 问题来源：archive_for_skill 24 模块全量实测（28 agent，260 资产）
> 核心理念：**零抽样全量阅读 / 1 PROJECT_ROOT = 1 套 12 资产 / 模块级全量执行 → 项目级聚合合成**

---

## 0. 为什么推倒 v3.5 重构

### 0.1 v3.5 的三个根本缺陷

| # | 缺陷 | 24 模块实测表现 | 根因 |
|---|------|------|------|
| 1 | 4 阶段管线假设单模块 ≤500 Java | synergy 1197 Java 单模块就跑不动 | 管线没有自我感知项目规模的能力——每 worker 分配到的文件数不做上限控制 |
| 2 | 上下文的唯一出路是**分层抽样** | 4 个 >500 Java 模块失速后被迫抽样 | "全量阅读"是核心承诺，抽样即是违背 |
| 3 | 多模块场景无原生支持 | 24 模块各产出一套 asset-docs——分散在子目录 | 方案从头到尾假设 1 PROJECT_ROOT = 1 模块 |

### 0.2 v3.6 的根本转向

```
v3.5: 1 个管线处理 1 个模块 → 大模块抽样 + 多模块拆散
v3.6: 模块级全量 worker + 项目级单 agent 合成 → 全量 + 单套
```

### 0.3 核心约束

| 约束 | 含义 |
|------|------|
| **零抽样** | 每个 Java 源文件必须被至少 1 个 worker 读取 |
| **1 PROJECT_ROOT = 1 套资产** | 不论项目有几个模块，`asset-docs/` 下只有 **1 套** 12 篇资产 |
| **每 worker ≤ 80 源文件** | 24 模块实测边界——超过则 agent stall |
| **单 agent 合成** | Phase 1 合成 agent 只读中间 JSON（~60KB / ~2000 行），不读源文件 |

## 1. 架构总览

### 1.1 两阶段新管线

```
Phase 0 · 模块级全量执行（可并行，跨 turn）
  │
  ├─ Step -1: 模块发现 + 清单生成
  │   find ${PROJECT_ROOT} -maxdepth 2 -name "pom.xml" → MODULE_COUNT
  │   生成 .module-manifest.json
  │
  ├─ Step 0: 按模块规模分配 worker
  │   每个模块：if java_count ≤ 80 → 1 worker, ≤ 300 → 2-3 worker, ≤ 800 → 4-8 worker, >800 → 8-15 worker
  │   每个 worker ≤ 80 个 Java 文件
  │   零抽样——所有文件都分配
  │
  ├─ Step 1: spawn worker（全量阅读）
  │   每个 worker 读 ≤ 80 个源文件
  │   产出 1 个 asset-docs/.intermediates/<module>/<asset_type>.json
  │   含 12 类结构化事实（entity / endpoint / SQL / risk / service / rule / antipattern / debt）
  │
  └─ Step 2: 编排者收集所有 worker 产出
       │
       ├─ 跑 Gate B: check-severity-alignment.sh（严重度对齐）
       ├─ 跑 Gate D: check-phase-facts.sh（phase-facts 完整性）
       └─ 跑 Gate C: verify-output-path.sh（输出路径正确）
       │
       ▼ 全部通过 → 进入 Phase 1

Phase 1 · 项目级单 agent 合成（单 turn，必须单 agent 完成）
  │
  ├─ Step 3: 编排者读取全部 .intermediates/*.json
  │   24 模块 × ~50 行/模块 = ~1200 行 JSON
  │   ≈ 40K tokens —— 远低于 200K 上限
  │
  ├─ Step 4: 按 12 篇资产模板逐篇合成
  │   2. 数据模型：全 project entity 汇总（去重 + 跨模块引用标记）
  │   3. API 清单：全 project endpoint 汇总
  │   4. Mapper 清单：全 project SQL 汇总
  │   5. 服务逻辑：全 project service 汇总
  │   6. 安全认证：全 project 风险汇总
  │   10. 业务流图：全 project 业务流
  │   11. 技术债：全 project 技术债
  │   12. 修复建议：基于 11 优先级排序
  │   13. 反模式扫描：全 project 反模式
  │   01. 系统总览：全局数字一览 + 架构决策推理
  │   00. 文档索引：阅读建议
  │
  ├─ Step 5: 跑 check-all.sh 校验
  │
  └─ Step 6: 清理 .intermediates/（过程产物），保留 asset-docs/
```

### 1.2 Phase 0 的 Worker 分配算法

```
输入：模块名、该模块 Java 文件数
输出：worker 分配方案（每个 worker 的文件列表）

算法：
1. if Java ≤ 80:  1 个 worker（全量，所有类型文件打包）
2. if Java ≤ 300: 按文件类型拆
   - worker-1: Controller + Config（安全扫描相关）
   - worker-2: Service + Mapper + Entity（数据 + 业务）
   = 2-3 worker，每个 ≤ 100 文件
3. if Java ≤ 800: 按包/子模块拆
   = 4-8 worker，每个 ≤ 80 文件
4. if Java > 800: 按包拆 + 细拆
   = 8-15 worker，每个 ≤ 80 文件
```

**关键**：所有情况都是全量阅读——只是拆的 worker 数不同。**没有抽样，没有分层**。

24 模块实测中 4 个 >500 Java 模块用分层抽样补救成功——但那是**战时手段**。v3.6 用"增加 worker 数"替代"降低覆盖率"。

### 1.3 中间产物格式（.intermediates/*.json）

每个 worker 输出一个 JSON 文件，不是 Markdown。JSON 便于 Phase 1 agent 程序化聚合——去重、合并、交叉引用。

```json
{
  "module": "devops-message",
  "worker_id": "W-03",
  "java_files_read": 42,
  "java_files_total": 125,
  "facts": {
    "entities": [
      {"class": "Handle", "table": "handle", "fields": ["handleId","userId"], "has_ddl": true, "ddl_file": "sql/devops-message-basic.sql"}
    ],
    "controllers": [
      {"class": "HandleController", "prefix": "/handle", "endpoints": 4, "auth": "none", "file": "HandleController.java"}
    ],
    "endpoints": [
      {"method": "GET", "path": "/handle/page", "controller": "HandleController", "auth": "none", "file_line": "HandleController.java:45"}
    ],
    "sql_operations": [
      {"type": "SELECT", "mapper": "HandleMapper", "table": "handle", "xml_file": "HandleMapper.xml:12", "injection_risk": false}
    ],
    "security_risks": [
      {"severity": "P0", "title": "Nacos密码明文", "file_line": "bootstrap.yml:11", "detail": "CxMtyx@9527. 默认值"}
    ],
    "services": [
      {"interface": "MessageService", "impl": "MessageServiceImpl", "methods": 8, "dependencies": ["HandleMapper"]}
    ],
    "business_rules": [
      {"rule": "发送失败不重试", "source": "SendMessageServiceImpl.java:89", "type": "error_handling"}
    ],
    "antipatterns": [
      {"tag": "@copy-paste", "severity": "P2", "location": "SendMessageServiceImpl", "count": 4}
    ],
    "tech_debts": [
      {"severity": "P1", "type": "hardcoded", "location": "application.yml:65"}
    ],
    "dependencies": [
      {"type": "feign", "from": "devops-message", "to": "devops-platform", "interface": "NoticeFeign"}
    ]
  }
}
```

**JSON 大小估算**：
- devops-message（125 Java）→ ~2KB JSON
- devops-synergy（1197 Java，12 个 worker）→ ~20KB JSON
- 24 模块合计 → ~60KB JSON

Phase 1 agent 读取 60KB JSON ≈ 2000 行 ≈ 40K tokens → **远低于 200K 上限**。

### 1.4 Phase 1 合成逻辑

```
编排者（单 agent，变体 D：Aggregate）：
1. 通读全部 .intermediates/*.json 的 facts 数组
2. 去重 + 合并：
   - 同一 entity 在多个模块出现 → 合并字段，标记 "跨模块: [module-a, module-b]"
   - 同一 security_risk 在多个模块出现 → 合并为 1 条，count += 1
3. 跨模块交叉引用：
   - dependencies 数组 → 构建全 project Feign 调用拓扑图
   - entity 跨模块一致性 → 同一 entity 名在不同模块中拼写不同 → 标记 WARN
4. 对每个资产类型（02/03/04/05/06/10/11/12/13）：
   a. 提取该类型的全部数据
   b. 按模板格式生成 Markdown 表格
   c. 写 asset-docs/<NN>-<name>.md
5. 01 系统总览：全局数字一览 + 架构决策推理 + 跨模块 Feign 拓扑图
6. 00 文档索引：阅读建议 + 跨模块资产清单
```

**为什么单 agent 就够了**：60KB JSON 是已提取的**结构化事实**——每个 entity 已经被 worker 读源码时整理成了字段列表 + 文件引用。合成 agent 的工作是"排序、去重、填表"——不是"读源码理解"。认知负担 ≈ 读 2000 行 text + 写 12 篇 Markdown。

## 2. Worker JSON Schema

`references/worker-json-schema.md` 定义完整的 JSON schema，每个 worker 必须遵守。

```markdown
# Worker JSON Schema

## 顶层结构
{
  "module": "<string>",          // 模块名（必填）
  "worker_id": "<string>",       // worker ID（必填，如 W-02）
  "java_files_read": <int>,      // 本 worker 实际读取的 Java 文件数（必填）
  "java_files_total": <int>,     // 本模块 Java 文件总数（必填，用于算覆盖率）
  "facts": { ... }               // 结构化事实（必填，至少一个字段非空）
}

## facts 字段规范
每个字段为数组。若本 worker 分配的文件中无对应类型，写空数组 []。
禁止写 null、禁止写字符串 "N/A"。

### entities
{ "class": "<必填>", "table": "<必填，无表写 ''>", "fields": ["...>"], "has_ddl": <bool>, "ddl_file": "<有DDL时必填>" }

### controllers
{ "class": "<必填>", "prefix": "<必填>", "endpoints": <int>, "auth": "<必填>", "file": "<必填>" }

### endpoints
{ "method": "<GET|POST|PUT|DELETE>", "path": "<必填>", "controller": "<必填>", "auth": "<必填>", "file_line": "<必填>" }

### sql_operations
{ "type": "<SELECT|INSERT|UPDATE|DELETE>", "mapper": "<必填>", "table": "<必填>", "xml_file": "<必填>", "injection_risk": <bool> }

### security_risks
{ "severity": "<P0|P1|P2|P3>", "title": "<必填>", "file_line": "<必填>", "detail": "<必填>" }

### services
{ "interface": "<必填>", "impl": "<必填>", "methods": <int>, "dependencies": ["...>"] }

### business_rules
{ "rule": "<必填>", "source": "<必填 file:line>", "type": "<必填>" }

### antipatterns
{ "tag": "<@sqlinjection|@secret-leak|...>", "severity": "<P0-P3>", "location": "<必填>", "count": <int> }

### tech_debts
{ "severity": "<P0-P3>", "type": "<hardcoded|dead_code|copy_paste|todo|deprecated>", "location": "<必填>" }

### dependencies
{ "type": "<feign|mq|rpc|db>", "from": "<必填>", "to": "<必填>", "interface": "<必填>" }
```

## 3. 5 个硬 Gate

| Gate | 时机 | 脚本 | 阻断条件 |
|------|------|------|---------|
| **A: Stall** | 每个 worker spawn 时 | agent-watchdog.sh（已有） | 300s 无输出 → kill + 该 worker 降级串行 |
| **B: 严重度** | Phase 0 全部 worker 完成后 | check-severity-alignment.sh（新增） | 同一 file:line 不同 P 级 → Phase 1 拒绝启动 |
| **C: 路径** | 每个 worker 返回时 | verify-output-path.sh（新增） | PATH_VERIFIED ≠ 预期路径 → 作废重跑 |
| **D: Facts** | Phase 0→1 过渡 | check-phase-facts.sh（新增） | .phase-facts.md 缺失/不完整 → 管线中止 |
| **E: 绑定** | v35-pipeline.sh 调用 | 脚本 BINDING: 前缀 | 编排者不可覆盖 worker 上限/模式路由 |

### Gate A: Agent Stall 自动处理

agent-watchdog.sh 升级为管线必带守护进程。编排者 spawn worker 时必须同时启动 watchdog。
Stall 触发 → kill stall agent + 标记 .module-status.json failed + 该模块降级为单 worker 串行。

### Gate B: 严重度对齐

Phase 0 所有 worker 完成后、Phase 1 启动前，跑 `check-severity-alignment.sh`。
逻辑：从所有 .intermediates/*.json 的 security_risks 中提取同一 file:line 的 severity 字段。
若同一位置在不同 worker 中有不同 P 级 → exit 1，输出冲突清单。
编排者人工裁决（取最高严重度），修正 JSON 后再跑检查。

### Gate C: 输出路径三重保护

Worker prompt 加三阶段验证：
1. 写前：声明 `WRITE_TARGET: <绝对路径>`
2. 写后：`ls -la <绝对路径>` 确认存在
3. 返回：摘要含 `PATH_VERIFIED: <实际路径>`

编排者收到后 grep PATH_VERIFIED，与预期 `${PROJECT_ROOT}/asset-docs/.intermediates/` 比对。

### Gate D: .phase-facts.md 强制校验

`check-phase-facts.sh` 验证 5 项：文件存在 / 非空 / 4 段齐全 / Phase≥2 有严重度对齐段 / 数字字段非空。
不通过 → 管线中止。

### Gate E: v35-pipeline.sh 输出绑定

脚本输出加 `BINDING:` 前缀（编排者不可覆盖）和 `ADVISORY:` 前缀（编排者可酌情调整）。
绑定项：模式路由决策 / worker 上限 / 10a+10b 拆分。

## 6. 开发计划（TDD + CI + Auto Review）

### 6.1 TDD 流程

每个新增脚本按以下顺序开发：

```
[1] Red    → 写 test_<script>.sh
             测试必须覆盖：正常通过 / 失败阻断 / 边界值
             先跑 → 确认 FAIL ✅

[2] Green  → 写最少量代码让 test_<script>.sh 通过
             bash -n 语法校验
             bash _proc-use/tests/test_<script>.sh → PASS ✅

[3] Refactor → 重构保持测试绿，不添加新功能
              bash _proc-use/tests/run-all.sh → 全部 PASS ✅
```

## 7. 回退点

| Phase | 回退方式 | 风险 |
|:---:|------|:---:|
| P1 | 新脚本不影响 v3.5 管线；删新文件即回退 | 低 |
| P2 | 合成脚本依赖 JSON schema；格式不兼容则回退 JSON 格式 | 中 |
| P3 | Gate 脚本独立运行不耦合管线；不通过时人工裁决 | 低 |
| P4 | git tag v3.5-final；回退 checkout tag | 低 |

**核心原则**：P1-P3 的新脚本和 v3.5 管线**并行**，不覆盖。P4 是破坏性变更加 tag 保护。
