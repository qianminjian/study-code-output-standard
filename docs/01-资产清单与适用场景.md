# 01 · 资产清单与适用场景

> 把抽象的方法论变成具体的产出物。**这一篇是整个标准的事实表**。

---

## 1. 资产全清单（12 篇）

| 编号 | 资产 | 必选性 | 输入来源 | 主要读者 |
|---|---|:-:|---|---|
| 00 | 文档索引 | 必选 | 所有其他资产 | 全部 |
| 01 | 系统总览 | 必选 | 项目元信息 + 顶层文件 | 全部 |
| 02 | 数据模型与表结构 | 必选 | ORM 实体 + Mapper + DDL | 后端 / DBA |
| 03 | API 接口清单 | 必选 | Controller / 路由文件 | 后端 / 前端 / 联调 |
| 04 | 数据访问操作清单 | 必选（关系型） | Mapper 接口 + SQL/XML | 后端 |
| 05 | 服务与业务逻辑 | 必选 | Service / 业务实现 | 后端 |
| 06 | 安全 / 认证 / AOP / Config | 必选 | Security / Filter / Config | 后端 / 安全 |
| 07 | 前端页面与组件清单 | 必选 | 视图 / 组件文件 | 前端 |
| 08 | 前端状态与路由 | 必选 | Store / Router / HTTP | 前端 |
| 09 | 静态站 / 多端 | 可选 | 静态 HTML / 小程序 | 前端 / 静态端 |
| 10 | 业务流图（端到端） | 必选 | 跨层调用链分析 | 全部 |
| 11 | 技术债与遗留项 | 必选 | 全文扫描 + TODO/反模式 | 架构师 / Tech Lead |

> 编号体系：固定不变。**不要为不同项目增减编号**——这会让模板失配。
> 某项目不涉及的领域，**保留编号但内容为"不适用"**。

## 2. 适用矩阵（按项目类型）

### 2.1 全栈 Web（推荐全套 12 篇）

**典型项目**：Spring Boot/Django/Express + Vue/React + 静态展示站
**资产选择**：00-12 全部
**估计工时**：1 个 AI + 1 人 1-2 周可完成

### 2.2 纯后端服务 / API

**典型项目**：Go/Python/Node 后端，无前端
**资产选择**：

| 必选 | 必选 | 必选 | 必选 | 必选 | 必选 | 必选 | — | — | 必选 | 必选 | 必选 |
|---|---|---|---|---|---|---|---|---|---|---|---|
| 00 | 01 | 02 | 03 | 04 | 05 | 06 | 07 | 08 | 10 | 11 | 12 |

**省略**：07-08 标记"前端由调用方实现，参见 OpenAPI 文档"，09 删除。

### 2.3 纯前端 SPA

**典型项目**：React/Vue/Angular + BaaS
**资产选择**：00, 01, 07, 08, **02'**(BaaS 数据模型), **03'**(BaaS API), 10, 11, 12

> **02' / 03' 表示后端即服务，资产化方式由 BaaS 决定（如 Supabase 直接消费其 schema JSON）**

### 2.4 移动 App（Native / RN / Flutter）

**资产选择**：

- 00, 01（含平台分支）, 02（业务模型）, 03（HTTP API）, 10, 11, 12
- 07 → 改为 **07-界面与页面清单**（按平台单独输出）
- 08 → 改为 **08-状态与导航**（含平台特定导航）
- 09 → 改为 **09-多端差异**（iOS / Android / 微信小程序 等）

### 2.5 微服务 / 多仓库

每个仓库独立产 12 篇，再额外加：

- `00-跨服务索引.md`：服务清单 + 服务间调用关系 + 数据所有权
- `12-跨服务治理.md`：链路追踪 / 限流降级 / 幂等 / 分布式事务

### 2.6 老旧系统（无构建工具 / jQuery / 无 ORM）

- 04 改为 **04-数据访问与存储**（直接查 SQL 文件、CSV、Excel、文件）
- 06 改为 **06-安全审计**（无 Spring Security 时改为手工审计）
- 08 改为 **08-状态与脚本**（无 Store 时改为 jQuery 全局变量/命名空间）

## 3. 适用矩阵（按项目规模）

| 规模 | 代码量 | 资产深度 | 估计工时 | 工具支持 |
|---|---|---|---|---|
| **微型** | < 1 万行 | 12 篇中 5-6 篇 | 1-2 天 | AI 全自动 |
| **小型** | 1-10 万行 | 12 篇完整 | 3-5 天 | AI + 人工校验 |
| **中型** | 10-50 万行 | 12 篇 + 跨服务索引 | 1-2 周 | 分阶段、AI 主导 |
| **大型** | 50 万+ 行 | 12 篇 + 多仓 + 治理 | 1 个月+ | 团队协作 |

> **微型项目**可以**压缩 02/03/04/05 为一篇 "后端能力清单"**，但**保留 12 篇编号体系**。

**强引用关系**（必须保持一致）：

- 02 ↔ 03（数据模型 ↔ API 入参出参）
- 02 ↔ 04（实体 ↔ Mapper）
- 03 ↔ 05（API ↔ Service）
- 03 ↔ 06（API ↔ 鉴权）
- 07 ↔ 08（组件 ↔ 状态 / 路由）
- 10 ↔ 02-09（业务流 ↔ 各层资产）
- 11 ↔ 02-10（技术债 ↔ 各层资产）
- 12 ↔ 11（修复建议 ↔ 技术债）

## 5. AI 视角的资产最小可消费集合

如果只给 AI 喂**3 篇**（最小可消费）：

1. **01-系统总览**（边界 + 技术栈 + 命名规范）
2. **02-数据模型**（实体关系）
3. **03-API 清单**（端点 + 入参出参 + 鉴权）

如果只给 AI 喂**5 篇**（推荐）：

加上 **05-服务与业务逻辑** + **11-技术债清单**。

如果给 AI 喂**全部 12 篇**：可以承担**新功能开发 + 重构 + 修 Bug** 三类任务。

## 6. 资产成熟度模型（4 级）

| 级别 | 名称 | 描述 | 验收 |
|---|---|---|---|
| **L1** | 草稿 | 12 篇都有初稿，但存在缺失、错引 | 完成 04-工作流第一步 |
| **L2** | 可消费 | 结构齐备、内容正确、严重度分级完整 | 通过 06-门禁所有 P0 项 |
| **L3** | 可维护 | 资产随代码同步演进（CI 校验） | 接入资产-代码一致性检查 |
| **L4** | 可治理 | 季度复盘 + 跨项目复用 + 资产即文档（TechDocs 化） | 接入 Backstage / 内部门户 |

**目标**：每个项目至少达到 **L2**。
