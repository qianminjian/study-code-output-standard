# 00 · 标准代码资产方法论 — 总览

> 适用对象：希望通过**读代码**反向沉淀出**对人和 AI 都可消费**的系统资产的项目。
> 不是 README 模板，不是架构图工具，是一套**让存量代码可被复用、可被维护、可被 AI 编程**的工程化标准。

---

## 1. 为什么需要这套资产

存量系统普遍面临四类痛点：

| 痛点 | 表现 | 后果 |
|---|---|---|
| **黑盒** | 接手人靠口口相传，文档过期、缺失或从未存在 | 维护成本指数增长，不敢动 |
| **不敢改** | 不知道哪些是核心、哪些是边角、哪些是炸弹 | 改一个 bug 引发三个 P0 |
| **AI 编程失效** | LLM 喂入整个仓库也只能给局部补丁 | "代码量大但 AI 帮不上忙" |
| **重复造轮子** | 每次都要重新摸一遍 | 同样的字段、同样的接口、同样的坑反复踩 |

**根本原因**：缺乏一套**从代码反向生成**、**对人和 AI 都结构化**、**覆盖全栈全生命周期**的资产。

本方法论的目标就是给出一套**产出这种资产的标准**。

---

## 2. 核心思想

三条原则贯穿全文：

### 2.1 资产分层（自上而下五层）

```
┌─────────────────────────────────────────────────────────────┐
│  L1 导航层       索引 + 读法         让人 / AI 一眼定位     │
│  L2 全局层       系统总览            边界 / 技术栈 / 关系图 │
│  L3 静态资产层   数据 / 接口 / 组件  按子系统横向切片       │
│  L4 行为层       业务流图（端到端）  按场景纵向串联         │
│  L5 诊断层       技术债 / 修复建议  为演进提供依据         │
└─────────────────────────────────────────────────────────────┘
```

每一层有明确的**输入**、**输出**、**质量门禁**、**AI 协作方式**。

### 2.2 资产可消费（双受众）

同一份资产对**人**和**AI**都需可消费：

| 维度 | 对人 | 对 AI |
|---|---|---|
| **可读** | 自然语言 + 表格 + 流程图 | 严格的 Markdown 结构 + 编号 + 标签 |
| **可定位** | 目录 + 命名空间 | 编号 + 元数据 + 交叉引用 |
| **可验证** | 自检清单 | 严重度等级 + 反模式标签 |
| **可复用** | 模板填空 | JSON Schema / 字段约束 |

**反例**：散文式架构文档 → 人爱看、AI 无法用 → 仅做手册。
**正例**：表格 + 编号 + 严重度 → 人爱看 + AI 可抽取 → 即可做手册又可做编程上下文。

### 2.3 资产可演进（带版本与责任人）

每篇资产带：

- `整理时间`、`数据来源`、`责任人`
- `最后验证 commit`、`严重度等级`
- `已知盲区`、`TODO`

资产**不是一次性交付**，是**随代码同步演进的活物**。

## 3. 在业界文档体系中的位置

本方法论**不取代**已有体系，而是**补足**：

| 体系 | 关注点 | 与本方法论关系 |
|---|---|---|
| **C4 Model** | 架构可视化（Context/Container/Component/Code） | 互补：把"画图"标准化为"出资产" |
| **arc42** | 架构文档模板（12 段） | 互补：本方法论继承其"分段 + 模板"思想 |
| **Diataxis** | 文档四象限（Tutorial/How-to/Reference/Explanation） | 互补：本方法论属于"Reference + Explanation"层 |
| **ADR** | 架构决策记录（why） | 互补：本方法论 04-修复建议可与 ADR 联动 |
| **OpenAPI / AsyncAPI** | API 规范 | 互补：本方法论 03-API 清单是 OpenAPI 的"摘要 + 人话版" |
| **OpenSSF SLSA / OWASP** | 安全基线 | 互补：06-安全认证 中嵌入 OWASP Top 10 检查 |
| **Conventional Commits / SemVer** | 代码版本 | 互补：本方法论用 commit hash 锚定资产版本 |

**与代码生成工具的关系**：

- 与 [Backstage](https://backstage.io/) 互补：本方法论输出 Markdown 资产，可被 Backstage TechDocs 渲染。
- 与 [Context7](https://context7.com/)、[Sourcegraph](https://sourcegraph.com/) 互补：这些是"代码即文档"的反方向（机器读代码），本方法是"代码 → 人话文档 → 喂回 AI"。

## 4. 适用边界

### ✅ 适用

- **存量项目接手**：2 个月以上、有真实代码、有维护需求
- **老系统现代化**：准备大改前需要"摸家底"
- **AI 编程上下文建设**：希望 AI 能基于项目资产稳定产出
- **新人入职 / 跨团队交接**：需要"代码层"以外的入口
- **架构治理 / 技术债盘点**：年度/季度演进规划

### ❌ 不适用（或简化处理）

- **全新项目从零起步**：资产先空着，按需补充
- **Demo / 1-2 周原型**：成本高于收益
- **库 / SDK / 单一工具**：OpenAPI 单一文档足够
- **强合规领域（金融/医疗/军工）**：需补强合规与审计资产（建议在 06-安全认证 之外另立）

### 🟡 部分适用

- **微服务 / 多仓库**：每个仓库一份资产，再在 `docs/00-索引.md` 统一索引
- **前端 SPA / 移动 App**：模板 07/08 适用，02/03/05 视情况简化
- **纯静态站 / 无后端**：只取 01/07/09/11

## 5. 与 AI 编程的结合点（核心卖点）

AI 能否写出**符合项目规范**的代码，取决于喂它的**上下文质量**。本方法论的资产就是这种上下文：

```
┌──────────────────┐    喂入    ┌──────────────────┐
│ 本项目 docs/ 资产 │ ────────► │  LLM 编程助手     │
│  - 02 数据模型     │           │  (Claude Code 等) │
│  - 03 API 清单     │           └──────────────────┘
│  - 10 业务流图     │                 │
│  - 11 技术债清单   │                 ▼
└──────────────────┘            符合规范的代码
```

**具体场景**：

| 场景 | 喂入资产 | AI 产出 |
|---|---|---|
| 新增 Controller | 02 + 03 + 06 + 10 | 一个符合鉴权、命名、流程规范的 Controller |
| 新增 Mapper | 02 + 04 + 11（命名不一致） | 一个修复了历史问题的 Mapper |
| 修复 P0 Bug | 11 + 12 | 优先按 12 修复建议执行 |
| 重构 Service | 05 + 10 + 11 | 一个无副作用的重构方案 |
| 编写单元测试 | 02 + 03 + 05 + 10 | 覆盖核心流程 + 边界条件的测试集 |

**反过来**，本方法论的反向阅读步骤（04）也是 AI 执行的：人给出 Prompt 模板（05），AI 跑出资产，**人做交叉校验**。

---

## 6. 资产 = 项目的"第二份代码"

> 一份代码是给机器执行的；
> 一份资产是给人和未来的 AI 阅读的；
> 二者必须保持**双向同步**，否则会像注释一样腐烂。

**核心方法论**：

> 改代码 = 改资产；改资产 = 改代码。资产是**代码的伴生品**而非附件。
