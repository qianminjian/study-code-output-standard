# 04 · 反向阅读工作流

> 把"看代码"变成"出资产"的可重复 6 步法。
> 适用：**人主导 + AI 辅助**（默认）或 **AI 主导 + 人校验**（小型项目）。

---

## 0. 流程总览

```
Step 0  启动确认 (Confirm)             ← 0.1d（Claude Code 中）
Step 1  勘探 (Reconnaissance)        ← 1-2 天
Step 2  分类 (Classification)        ← 0.5 天
Step 3  抽取 (Extraction)            ← 3-7 天（AI 重头戏）
Step 4  校验 (Validation)            ← 1-2 天
Step 5  反模式深度标注 (Annotation)   ← 1-2 天（v2.2 派生 13 号资产）
Step 6  派送 (Delivery)               ← 0.5 天
Step 7  回写 (Writeback)               ← 0.5d（v2.2 新增 · 闭环）
```

**总周期**：1-2 周（小型）/ 2-4 周（中型）/ 1 个月+（大型）
> v2.2 升级为 7 步法（新增 Step 0 启动确认 + Step 7 回写闭环）

---

## Step 1 · 勘探（Reconnaissance）

**目标**：建立项目"心智地图"，确定 12 篇资产的覆盖范围与变体。

### 1.1 必做动作

- [ ] 跑项目主干（按 01-总览的"启动顺序"），确认能跑起来
- [ ] 用 `cloc` / `tokei` 统计代码量
- [ ] 用 `tree -L 3` 拿到顶层目录
- [ ] 列 `package.json` / `pom.xml` / `requirements.txt` 拿到依赖
- [ ] 找到 README、ARCHITECTURE、CHANGELOG（**有就参考，没有就当不存在**）
- [ ] 找入口文件（`main` / `Application` / `index`）

### 1.2 必出产物

- **项目骨架笔记**（半页纸），含：
  - 启动命令
  - 关键路径（每个子模块入口）
  - 主要依赖
  - 已知"看到一半就要警惕"的区域

### 1.3 AI 协作方式

**Prompt 模板**（见 05-Prompt 工具集 §1）：

```
我给你一个 <项目名>，请执行 Step 1 勘探：
- 跑 `tree -L 3` 看目录
- 读 README、pom.xml、package.json
- 找到 5 个最关键的入口文件
- 输出：项目骨架笔记
```

### 1.4 退出条件

- 团队新人拿到骨架笔记**能自己跑起来**

## Step 2 · 分类（Classification）

**目标**：把项目映射到 12 篇资产框架，识别**变体**（如多端、纯前端、纯后端）。

### 2.1 必做动作

- [ ] 套用 01-资产清单的适用矩阵
- [ ] 决定命名变体（单体？前后端分离？多端？）
- [ ] 决定省略哪些（如纯后端省略 07/08/09）
- [ ] 决定 12 篇是否需要"占位文件"（不适用时保留编号）

### 2.2 决策表

| 现象 | 决策 |
|---|---|
| 有 `views/` 和 `components/` | 07 完整输出 |
| 只有一个 `app.js` | 07 改"页面与脚本" |
| 有静态 HTML 站 | 09 完整输出 |
| 后端用 Spring Boot + MyBatis | 04 完整输出 |
| 后端用 Django ORM | 04 改"ORM 操作清单" |
| 前端用 Vue 2 | 08 含 Vuex |
| 前端用 Vue 3 | 08 含 Pinia |
| 前端用 React | 08 含 Redux/Zustand |
| 移动 App（iOS+Android） | 09 拆 09-iOS / 09-Android |
| 微服务多仓 | 加 13-跨服务索引 |

### 2.3 必出产物

- **资产清单决策表**（一页）
- **占位文件**：不适用也保留（如 09-静态前台.md 写"本项目无静态站"）

### 2.4 退出条件

- 12 篇资产都明确"输出" / "省略" / "占位"

## Step 3 · 抽取（Extraction）— AI 主战场

**目标**：从代码生成 12 篇资产的初稿。

### 3.1 抽取顺序

**关键原则**：**先静态后行为，先具体后综合**。

```
1. 02-数据模型与表结构 ← ORM 实体 + DDL（最具体）
2. 03-API 清单 ← Controller / Route（最稳定）
3. 04-Mapper 清单 ← 依赖 02
4. 06-安全认证 ← 单独扫（优先级高）
5. 05-服务与业务逻辑 ← 依赖 02/03/04
6. 07-前端页面与组件 ← 单独扫
7. 08-前端状态与路由 ← 依赖 07
8. 09-静态站 ← 单独扫
9. 10-业务流图 ← 跨 02-09 串联
10. 11-技术债 ← 跨 01-10 累积
11. 12-修复建议 ← 派生 11
12. 01-系统总览 ← 总结所有
13. 00-文档索引 ← 最后写
```

> **为什么 01 最后写**：01 是总结，必须等所有子资产完成。

### 3.2 每篇资产的抽取流程

以 03-API 清单为例：

```
a. AI 扫描所有 Controller
   - 文件列表
   - 注解 (@RequestMapping / @GetMapping)
   - 鉴权 (@PreAuthorize / @RequiresPermissions)
   - 公开 (@CrossOrigin / permitAll)

b. AI 生成表格草稿
   | 方法 | 路径 | 入参 | 鉴权 | 公开 | 说明 |

c. AI 补充统计
   - 端点总数
   - Controller 总数
   - 公开端点数
   - 权限串全集

d. AI 标注问题
   - tags 错位
   - 权限错用
   - 公开端点过多

e. 人校验（关键步骤）
   - 对照 Swagger 页面
   - 对照 Postman
   - 对照前端调用
```

### 3.3 AI Prompt 模式

**通用 Prompt 三段式**（每篇资产一个）：

```
【背景】
<项目类型>，代码根目录 <路径>，技术栈 <列表>。

【任务】
按 docs/03-文档模板与质量标准.md 的"03-API 清单"模板，
生成 docs/03-后端-Controller接口清单.md 初稿。

【输入文件】
- 路径：src/main/java/.../controller/**
- 关注：@RequestMapping、@PreAuthorize、@CrossOrigin

【输出格式】
严格按模板字段。表格列：方法、路径、入参、鉴权、公开、说明。
权限串全集单独表。公开端点单独表。

【质量要求】
- 端点总数 = grep -c "@(Get|Post|Put|Delete)Mapping" 数量
- 路径含全局前缀
- 每个端点有鉴权标注

【反例】
- 不写"见代码"代替入参
- 不省鉴权
- 不漏 Swagger tags 错位
```

完整 Prompt 见 `ai-prompts/` 目录。

### 3.4 必出产物

- 12 篇初稿（含占位）
- `ai-prompts/` 执行日志（哪些 Prompt 跑了、产物路径）

### 3.5 退出条件

- 12 篇都有"非空"内容
- 端点数 / 类数 / 文件数 等可验证数字齐全

## Step 4 · 校验（Validation）

**目标**：确保资产**与代码事实一致**。

### 4.1 三类校验

> ⚠️ **必须人工抽样 5-10 个端点 + 3-5 个业务流**，否则视为半成品——脚本只发现候选名单，最终正确性靠人。

#### 自动校验（CI / 脚本）

- 端点数 = grep 数
- Controller 数 = 文件数
- 权限串全集 = @PreAuthorize 集合
- 元信息头字段完整性
- 内部链接有效性
- 严重度字段值在 P0-P3 范围

#### 抽样校验（人）

抽 5-10 个端点：
- 对照 Swagger / Postman
- 跑一遍实际请求
- 看返回是否符合

抽 3-5 个业务流：
- 按 10-业务流图 走一遍
- 看是否漏掉异常分支

#### 同行校验（另一人）

- 找熟悉项目的人过一遍
- 重点看"是否漏了关键资产"

### 4.2 校验脚本骨架

```bash
#!/usr/bin/env bash
# docs/scripts/validate.sh
set -e

# 1. 元信息头
for f in docs/*.md; do
  grep -q "@meta" "$f" || { echo "MISSING META: $f"; exit 1; }
done

# 2. 端点数校验
expected=$(grep -r -c "@\(Get\|Post\|Put\|Delete\)Mapping" src/ | wc -l)
documented=$(grep -c "| \(GET\|POST\|PUT\|DELETE\) |" docs/03-*.md || true)
[ "$expected" -eq "$documented" ] || { echo "ENDPOINT MISMATCH: $expected vs $documented"; exit 1; }

# 3. 严重度字段
grep -E "P[4-9]" docs/ -r && { echo "INVALID SEVERITY"; exit 1; }

echo "OK"
```

### 4.3 必出产物

- **校验报告**（`docs/VALIDATION.md`），含通过项 / 失败项 / 修复记录

### 4.4 退出条件

- 自动校验 100% 通过
- 抽样校验 90%+ 通过
- 严重问题（错端点、错权限、错路径）= 0

## Step 5 · 反模式深度标注（Annotation · v2.2 改名 + 派生 13 号资产）

**目标**：把代码的"问题"与"风险"显式标出，**派生独立的 13-反模式扫描报告**（不再只写到 11-技术债清单）。

### 5.1 三类标注

#### 严重度标注（必备）

每个问题带 🔴P0 / 🟡P1 / 🟢P2 / ⚪P3。

#### 位置标注（必备）

格式：`<文件路径>:<行号>` 或 `<文件路径>:<行号范围>`。

#### 反模式标签（可选但推荐）

预定义标签集：

| 标签 | 含义 |
|---|---|
| `@sqlinjection` | SQL 注入风险 |
| `@secret-leak` | 密钥/密码明文 |
| `@cors-wildcard` | CORS 通配 |
| `@perm-mismatch` | 权限字符串错用 |
| `@admin-bypass` | 绕过 admin 保护 |
| `@null-pointer` | 空指针风险 |
| `@n+1` | N+1 查询 |
| `@dead-code` | 死代码 |
| `@hardcoded` | 硬编码 |
| `@wrong-default` | 默认值错误 |
| `@xss` | XSS 风险 |
| `@csrf` | CSRF 风险 |
| `@missing-tx` | 缺事务 |
| `@todo-stub` | TODO 占位未实现 |

**反模式标签用 Markdown 链接形式**：

```markdown
🔴 P0 `@sqlinjection` `NewsInfoMapper.xml#listAll` 使用 `${news_title}` 字符串拼接
```

### 5.2 必出产物（v2.2 拆分 11 / 13）

- **`11-技术债与遗留项.md`**：只收"**影响业务且有修复方案**"的反模式（如 @sqlinjection / @secret-leak / @cors-wildcard），带位置 + 严重度 + 修复要点
- **`12-修复建议与优先级.md`**：从 11 派生，**必须**带修复方案 + 工作量 + SLA
- **`13-反模式扫描报告.md`（v2.2 新增）**：收"**模式分布 / 扫法 / 降级建议**"，按 24 个标签分类，含统计与风险雷达

> **为什么拆 13 号**：原 11-技术债清单被 24 个反模式标签淹没（典型 30+ 条），不易聚焦业务问题。13 号报告专注"反模式本身"——标签分布 / 严重度热图 / 降级建议。

### 5.3 退出条件

- 11 含 ≥ 20 条标注问题（聚焦业务）
- 12 含 ≥ 10 条修复建议
- 13 含 24 标签全扫描结果（即便为 0 也列）
- 严重度分布合理（P0 < 10%）

## Step 6 · 派送（Delivery）

**目标**：让资产**可发现、可消费、可演进**。

### 6.1 必做动作

- [ ] 写 `CODEBUDDY.md` / `CLAUDE.md` 引用本资产
- [ ] 接 CI：`${SKILL_HOME}/scripts/check-all.sh` 在 PR 必跑
- [ ] 发布到 Backstage / GitBook / VitePress（如有内部平台）
- [ ] 通知团队"资产 v1.0 已就绪"
- [ ] 列入新人入职路径

### 6.2 演进机制

| 触发条件 | 动作 |
|---|---|
| PR 改 Controller | CI 跑 `check-all.sh`，人工同步 03 |
| PR 改 Mapper | 同步 02/04 |
| PR 修 P0 | 同步 11/12（标记"已修复"） |
| 季度 | 资产 review，重排严重度 |
| 大版本 | 资产同步 bump 主版本 |

### 6.3 必出产物

- 项目级 AI 引导文件
- CI 校验脚本
- 演进 SLA（写在 `docs/CHANGELOG.md`）

### 6.4 退出条件

- 资产**对外可消费**（新人 + AI 都能用）
- 资产**对内可维护**（CI + SLA）

## Step 7 · 回写（Writeback · v2.2 新增 · 闭环）

**目标**：让"**改代码 = 改资产；改资产 = 改代码**"金句真正落地——资产不是一次性附件。

**为什么新增本步**：原 6 步法只到 Step 6 派送，缺少"代码改完回头刷资产"机制，导致资产很快与代码漂移。Step 7 强制把回写变成 SLA。

### 7.1 必做动作

1. **改代码 → 同步刷资产**：
   - 改 Controller → 同步 `03-Controller接口清单`
   - 改 Mapper → 同步 `02-数据模型` + `04-Mapper操作清单`
   - 修 P0 → 同步 `11-技术债`（标"已修复"）+ `12-修复建议`（标"已完成"）
   - 改 Service 业务流 → 同步 `05-服务与业务` + `10-业务流图`

2. **CI 接入**：
   ```yaml
   # .github/workflows/docs-validate.yml
   on:
     pull_request:
       paths: ['src/**', 'asset-docs/**']
   jobs:
     validate:
       runs-on: ubuntu-latest
       steps:
         - uses: actions/checkout@v4
         - run: bash $SKILL_HOME/scripts/check-all.sh
   ```

3. **last_updated 时效校验**：
   - `check-meta.sh` 每日检查 12 篇 `last_updated` 距今 > 30 天 → 告警（v2.2 新增）
   - 不阻断 PR，但要求 owner 在合并前刷新

4. **版本对齐**：
   - 代码 bump 大版本（v2 → v3）→ 资产同步 bump（同步改 12 篇 `version` + CHANGELOG）
   - 季度：资产 review，重排严重度

### 7.2 必出产物

- `.github/workflows/docs-validate.yml`（CI 配置）
- 12 篇 `last_updated` 都新鲜（< 30 天）
- 团队 owner 接受"改代码 = 改资产"SLA

### 7.3 退出条件

- CI 接入完成（PR 必跑 `check-all.sh`）
- `last_updated` 30 天告警机制就位
- 资产与代码漂移时间 < 7 天

## 常见反模式

| ❌ 反模式 | ✅ 正解 |
|---|---|
| 一上来就让 AI 写 01-总览 | 01 最后写 |
| 12 篇并行抽 | 按依赖顺序串行 |
| 不写元信息头 | 强制 frontmatter |
| 端点数"约 70 个" | grep 数清楚 |
| 散文式描述 | 表格化 |
| 严重度自定义（"中"） | 统一 P0-P3 |
| 资产写完就扔 | 接入 CI 校验 |
| AI 写完不校验 | 人必抽 5-10 处 |
